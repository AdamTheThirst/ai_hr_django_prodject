# Веб-тренажёр обратной связи

## 1. Что это за проект

Веб-приложение на Django для тренировки навыка обратной связи в формате ролевого диалога с ИИ.

Пользователь:
- регистрируется в системе;
- выбирает игру;
- выбирает конкретный сценарий внутри игры;
- ведёт диалог с ИИ-персонажем в интерфейсе, визуально похожем на современные веб-чаты;
- завершает диалог вручную кнопкой «Дай обратную связь» или автоматически по таймеру;
- получает развёрнутый анализ своего диалога по набору аналитических критериев.

Проект является полной веб-заменой Telegram-версии. При этом архитектура должна быть спроектирована так, чтобы в будущем один и тот же Django-бэкенд мог обслуживать:
- веб-интерфейс;
- подключаемый Android/APK-клиент.

## 2. Назначение проекта

Основная задача проекта — дать пользователю безопасную среду для практики сложных разговоров и получения структурированной обратной связи по качеству собственной коммуникации.

Проект должен поддерживать:
- несколько игр;
- несколько сценариев внутри одной игры;
- один общий набор аналитических промтов на всю игру;
- возможность расширения без переделки ядра.

Пример логики:
- игра: «Я-высказывание»;
- сценарии внутри игры: «Подчинённый», «Коллега», «Руководитель»;
- все сценарии этой игры анализируются одним и тем же набором аналитических промтов.

## 3. Текущий функциональный объём первой версии

В первую версию входят:
- регистрация пользователя по никнейму, email и паролю;
- вход в систему и выход из системы;
- восстановление пароля по email;
- главная страница с выбором игры и кнопками запуска сценариев;
- страница игрового диалога;
- автоматический или ручной переход к анализу;
- страница результатов анализа;
- экспорт результатов в PDF;
- административный интерфейс;
- системные настройки, тексты интерфейса, игры, сценарии, промты и параметры LLM, редактируемые через админские интерфейсы.

В первую версию не входят как полноценные модули, но должны быть предусмотрены архитектурно:
- личный кабинет пользователя;
- энциклопедия.

На пользовательском интерфейсе для них показываются заглушки.

## 4. Основной пользовательский сценарий

1. Пользователь открывает сайт.
2. Регистрируется или входит в систему.
3. Попадает на главную страницу.
4. Выбирает игру.
5. Нажимает кнопку одного из сценариев.
6. Открывается страница чата.
7. В верхней части страницы фиксированно отображаются условия сценария (`conditions`).
8. ИИ автоматически начинает игру стартовым приветствием, заданным для конкретного сценария.
9. Пользователь пишет сообщения в чат.
10. Диалог продолжается в рамках одной страницы без её полной перезагрузки.
11. Пользователь либо нажимает «Дай обратную связь», либо ждёт завершения таймера.
12. Если пользователь закрыл вкладку или ушёл со страницы, диалог считается незавершённым.
13. Если в диалоге была хотя бы одна пользовательская реплика, запускается анализ.
14. Пользователь получает страницу результатов с карточками по каждому критерию и общей суммой баллов.
15. Пользователь может экспортировать результат в PDF.

## 5. Ключевые продуктовые правила

### 5.1. Игры и сценарии

- Игра — это верхний уровень логики тренировки.
- Сценарий — конкретная ситуация внутри игры.
- Одна игра может содержать несколько сценариев.
- У всех сценариев одной игры общий набор аналитических промтов.
- Игры, сценарии, тексты условий, стартовые реплики персонажей и аналитические промты редактируются через админский интерфейс.

### 5.2. Диалог

- Одновременно у пользователя может быть только один активный диалог.
- Диалог проходит в чат-интерфейсе без полной перезагрузки страницы.
- Поле условий закреплено в верхней части экрана.
- В интерфейсе есть только чат и кнопка «Дай обратную связь».
- Дополнительно на странице есть таймер.
- ИИ видит весь накопленный диалог, а не только последние сообщения.
- В первой версии поддерживается только текст.
- Возможность голосовых сообщений должна быть предусмотрена архитектурно как будущая заглушка.

### 5.3. Завершение диалога

Диалог может завершиться тремя способами:
- пользователь нажал кнопку «Дай обратную связь»;
- истёк таймер;
- пользователь закрыл вкладку, покинул страницу или оборвал сессию.

Правила завершения:
- по умолчанию длительность диалога — 10 минут;
- супер-администратор может менять глобальное значение таймера через админку;
- в будущем пользователь сможет в личном кабинете выбирать свой таймер в диапазоне 5–20 минут;
- при истечении времени пользователю показывается сообщение «Время диалога истекло» на 3 секунды, после чего он переводится на экран анализа;
- если пользователь покинул страницу, диалог помечается как «не завершён»;
- если пользователь не отправил ни одной реплики, анализ не запускается;
- если пользователь отправил хотя бы одну реплику, анализ запускается даже для незавершённого диалога.

### 5.4. Анализ

- Аналитика строится на отдельном наборе аналитических промтов.
- Каждый аналитический промт оценивает один критерий.
- У игры может быть много аналитических промтов; их количество может меняться.
- Для каждого критерия сохраняются:
  - текст анализа;
  - целочисленный балл.
- Общая сумма по всем критериям пользователю показывается, но в БД отдельно не хранится.
- Максимум суммы зависит от количества аналитических промтов и считается динамически.
- Результаты показываются на одной странице в виде карточек.
- В верхней части результата показывается сумма вида «N из M».
- В первой версии анализ запускается синхронно.
- Если ответ LLM для аналитики невалиден или не соответствует ожидаемому JSON-формату, система должна уметь корректно обработать ошибку, залогировать её и выполнить предусмотренную стратегию повтора/обработки.

### 5.5. PDF-экспорт

Экспортируемый PDF должен включать:
- название игры;
- название сценария;
- дату и время;
- статус завершения;
- общую сумму баллов в формате «N из M»;
- карточки по каждому критерию;
- полный транскрипт диалога.

## 6. Роли и доступы

### 6.1. Обычный пользователь

Может:
- регистрироваться и входить в систему;
- редактировать свой никнейм;
- запускать сценарии;
- вести диалог;
- получать анализ;
- экспортировать результат в PDF;
- использовать будущий личный кабинет после его реализации.

### 6.2. Администратор

Может:
- создавать свои игры;
- создавать и редактировать свои сценарии;
- управлять связанными с ними сущностями в пределах собственных прав.

Не может:
- изменять или удалять игры и сценарии, созданные другими администраторами;
- изменять глобальные UI-тексты;
- выполнять действия уровня супер-администратора.

### 6.3. Назначенный супер-администратор

Может:
- редактировать глобальные UI-тексты;
- редактировать системные настройки;
- управлять параметрами LLM;
- редактировать промты;
- управлять всеми играми и сценариями вне зависимости от автора;
- изменять и обновлять seed-данные пользователей, если это разрешено логикой интерфейса.

Не может:
- создавать новых администраторов.

### 6.4. Главный супер-администратор

Может всё, что может назначенный супер-администратор, и дополнительно:
- создавать новых администраторов;
- назначать и отзывать права супер-администратора;
- быть единственным источником создания административных ролей верхнего уровня.

## 7. UI и UX

### 7.1. Общие требования

- Подход: mobile first.
- Интерфейс должен корректно выглядеть и на десктопе.
- Стиль: минималистичный.
- Базовый визуальный ориентир: Material Design по принципам, но реализация — на Bootstrap 5 и локальном CSS.
- Требование: минимум JavaScript, только там, где он необходим для работы чата и динамических действий.

### 7.2. Главная страница

На главной странице должны располагаться:
- заголовок с названием игры;
- кнопки сценариев внутри игры;
- кнопка перехода к личному кабинету-заглушке;
- кнопка перехода к энциклопедии-заглушке;
- кнопка «Выйти».

### 7.3. Страница диалога

- Верхняя часть экрана содержит закреплённый блок условий сценария.
- Ниже идёт чат-лента.
- После открытия страницы персонаж начинает разговор автоматически.
- Во время ожидания ответа ИИ в чат вставляется визуальный индикатор «думаю...» в стиле современных LLM-чатов.
- Ответ ИИ подгружается в текущую страницу без полной перезагрузки.
- Стриминг ответа по токенам в первой версии не требуется.
- На странице есть поле ввода сообщения и кнопка отправки.
- Отдельно есть кнопка «Дай обратную связь».

### 7.4. Аватар пользователя

Вместо графической иконки пользователя используется:
- первая буква никнейма;
- на детерминированном монохромном светлом фоне;
- цвет закрепляется за пользователем;
- запрещены белый, чёрный, серый и ядовитые оттенки;
- фон подбирается так, чтобы чёрная буква всегда оставалась читаемой.

### 7.5. Иконки

- У каждой кнопки должно быть зарезервировано место под иконку.
- Базовый размер контейнера под иконку: 24x24 px.

## 8. Технологические решения

### 8.1. Базовый стек

- Backend: Django.
- Frontend: Django templates.
- UI: Bootstrap 5 + локальный CSS.
- JS: минимальный объём vanilla JS.
- База данных первой версии: SQLite.
- Хранение медиа: локально, в `MEDIA_ROOT` рядом с проектом.
- Работа с LLM: OpenAI-compatible client к внешнему vLLM endpoint.

### 8.2. Принципы архитектуры

- Один Django-монолит.
- Бизнес-логика должна быть отделена от представлений и готова к дальнейшему подключению `/api/v1/...`.
- Система должна быть спроектирована так, чтобы в будущем подключить Android/APK-клиент без переписывания ядра.
- Все основные связи в данных должны быть реализованы через корректные `ForeignKey`.
- Изменения схемы БД — только через Django migrations.
- Для мастер-данных должна использоваться архивация вместо жёсткого удаления там, где это не должно ломать исторические данные.

## 9. LLM-интеграция

Система использует OpenAI-совместимый клиент для обращения к vLLM-серверу.

Базовый ориентир подключения:

```python
from openai import OpenAI

client = OpenAI(
    base_url="https://...",
    api_key=""
)

chat_response = client.chat.completions.create(
    model="Qwen/Qwen3-32B",
    messages=[
        {"role": "system", "content": "Вы - роль"},
        {"role": "user", "content": "Сообщение пользователя"}
    ],
    max_tokens=200,
    temperature=0.7,
    top_p=0.8
)
```

Требования:
- одна и та же модель используется и для игрового диалога, и для аналитики;
- различие достигается только промтами;
- параметры модели редактируются супер-администратором через админку;
- у каждого параметра в админке должны быть пояснения: что он делает, минимальные и максимальные значения;
- система должна ограничивать длину пользовательского сообщения;
- значение по умолчанию для длины пользовательского сообщения: 2500 символов;
- это значение также должно быть редактируемым супер-администратором;
- система должна корректно обрабатывать ошибки LLM, делать предусмотренный повтор, показывать дружелюбное сообщение пользователю и обязательно логировать инцидент.
