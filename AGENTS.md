# AGENTS.md

## Назначение
Этот репозиторий описывает и реализует Django-монолит для веб-тренажёра обратной связи.

Цель агента:
- реализовывать проект строго по спецификациям репозитория;
- не выдумывать поведение, роли, экраны, поля моделей, API и бизнес-правила;
- вносить небольшие, проверяемые изменения;
- сохранять совместимость с будущим подключением нативного Android-клиента к тому же бэкенду.

## Язык проекта
- Интерфейс: русский.
- Контент промтов: русский.
- Комментарии в коде: русский.
- Docstring у каждого класса и каждой функции: обязательно на русском.

## Обязательное правило по docstring
Для каждого класса, метода, функции и сервиса нужен подробный docstring на русском языке.

Минимум, который обязан быть в docstring:
- что делает сущность;
- в каком контексте используется;
- какие параметры принимает;
- что возвращает;
- какие исключения или особые случаи возможны;
- какие побочные эффекты есть, если они есть.

Запрещено:
- оставлять классы и функции без docstring;
- писать формальные пустые docstring вроде «Функция делает действие»;
- писать docstring на английском, если нет специальной причины.

## Источники истины
Перед началом любой работы обязательно прочитать и использовать как основную спецификацию:
- README.md
- docs/PRODUCT_SPEC.md
- docs/USER_FLOWS.md
- docs/FUNCTIONAL_REQUIREMENTS.md
- docs/NON_FUNCTIONAL_REQUIREMENTS.md
- docs/ARCHITECTURE.md
- docs/DATA_MODELS.md
- docs/PROMPTS_SPEC.md
- docs/API_SPEC.md
- docs/ADMIN_PANEL_SPEC.md
- docs/TEST_CASES.md

Если документы противоречат друг другу:
- не придумывать компромисс самостоятельно;
- остановиться;
- явно перечислить противоречие;
- предложить минимально инвазивный вариант разрешения, но не внедрять его без явного указания.

## Границы реализации
Нужно соблюдать:
- Django-монолит;
- mobile first;
- Django templates + Bootstrap 5 + локальный CSS;
- минимум JavaScript;
- SQLite для V1;
- правильные связи через ForeignKey там, где это указано в спецификации;
- миграции только через Django migrations;
- архивирование вместо жёсткого удаления для мастер-данных, где это предусмотрено.

Не делать без прямого указания:
- SPA-фронтенд;
- React/Vue как основной фронтенд;
- PostgreSQL как обязательную БД для V1;
- WebSocket/стриминг токенов, если это отдельно не запрошено;
- функции, которых нет в спецификациях.

## Стиль разработки
Предпочтения по коду:
- явная и читаемая структура приложений;
- бизнес-логика — в сервисах, а не во view, где это возможно;
- тонкие view;
- формы, валидаторы и permissions — отдельно и прозрачно;
- минимум магии;
- предсказуемые имена файлов, классов, методов и шаблонов;
- без дублирования логики.

При изменении модели данных:
- обновлять модели и миграции вместе;
- не редактировать старые миграции задним числом, если нет отдельного указания;
- не ломать уже согласованные идентификаторы, статусы и связи.

## Правила по UI
Нужно соблюдать:
- минималистичный интерфейс;
- визуально аккуратный mobile first layout;
- хороший вид на desktop без отдельной сложной версии;
- чат без перезагрузки страницы;
- минимум JavaScript, только по необходимости;
- условия сценария закреплены в верхней части экрана чата;
- индикатор «думаю...» отображается в окне чата;
- место под иконки у кнопок должно сохраняться.

Не менять без указания:
- базовую структуру экранов;
- пользовательский поток;
- названия ролей;
- логику одного активного диалога.

## Правила по ролям и доступам
Роли и ограничения должны строго следовать спецификациям.

Нельзя:
- расширять права обычного администратора сверх описанного;
- давать назначенному супер-админу право создавать новых администраторов, если это не указано явно;
- давать обычному администратору возможность редактировать или удалять чужие игры, сценарии и связанные сущности;
- давать обычному администратору возможность менять UI-тексты.

## Правила по AI-части
Нужно соблюдать:
- использовать OpenAI-compatible client для vLLM;
- отделять игровые промты, аналитические промты и служебные промты;
- не смешивать opening message персонажа с системным игровым промтом, если в спецификации указано отдельное поле;
- учитывать, что один набор аналитических промтов принадлежит игре, а не отдельному сценарию;
- анализ запускать только если была хотя бы одна пользовательская реплика.

## Правила по безопасности и надёжности
Обязательно:
- серверная валидация входных данных;
- защита от двойной отправки;
- CSRF там, где она должна быть;
- логирование ошибок и важных технических событий в БД;
- аккуратная обработка ошибок LLM и JSON-ответов анализа.

Нельзя:
- отключать защитные механизмы ради упрощения;
- хардкодить секреты в коде;
- использовать dev-пароли как production-настройку.

## Seed-данные
Для локальной разработки и тестов должны существовать dev seed-учётки, если это предусмотрено спецификацией.

Важно:
- такие учётки считаются только dev/test-данными;
- в production они не должны создаваться автоматически без отдельного указания;
- логика создания seed-данных должна быть прозрачной и управляемой.

## Как выполнять задачи
Работай поэтапно.
Одна задача = один ограниченный блок реализации.

Под «одним блоком» понимается не «один markdown-файл документации», а один модуль или один аспект системы, например:
- только модели и миграции;
- только auth;
- только чатовый экран;
- только LLM-сервис;
- только PDF-экспорт;
- только admin panel.

Нельзя в одной итерации без явной просьбы одновременно:
- перестраивать модели;
- менять UI;
- внедрять LLM;
- переписывать роли;
- добавлять админку;
- менять API-контракты.

Если задача сформулирована слишком широко:
- сначала разбей её на этапы;
- предложи безопасный порядок;
- реализуй только тот этап, который прямо запрошен.

## Что нужно указывать в ответе после изменений
После каждой завершённой итерации нужно кратко и предметно указать:
- какие файлы изменены;
- что именно реализовано;
- какие миграции добавлены;
- какие тесты добавлены или обновлены;
- какие команды нужно выполнить локально;
- что пока не реализовано в рамках текущего этапа.

## Тесты
При реализации новой логики нужно добавлять или обновлять тесты там, где это разумно и не противоречит этапу.

Минимум:
- smoke для основных маршрутов и ключевых сервисов;
- тесты бизнес-ограничений, если они были реализованы;
- тесты прав доступа для ролей, если менялась авторизация или админская часть.

## Работа с документацией
Если изменение кода требует обновления документации репозитория:
- обнови соответствующий .md-файл;
- перечисли это в итоговом отчёте.

Если пользователь прямо просит реализовать код строго по уже согласованным спецификациям:
- не переписывай спецификации без необходимости;
- код должен подчиняться документации, а не наоборот.
