# TEST_CASES.md

## 1. Назначение документа

Документ фиксирует основные тестовые сценарии и edge cases для первой версии веб-тренажёра обратной связи.

Цели документа:
- дать команде и Codex понятный перечень проверок для реализации и приёмки;
- зафиксировать минимальный объём smoke-, integration- и regression-проверок;
- покрыть критические пользовательские и административные потоки;
- отдельно выделить пограничные случаи и ошибки интеграции с LLM.

Документ опирается на:
- `README.md`;
- `PRODUCT_SPEC.md`;
- `USER_FLOWS.md`;
- `FUNCTIONAL_REQUIREMENTS.md`;
- `NON_FUNCTIONAL_REQUIREMENTS.md`;
- `ARCHITECTURE.md`;
- `DATA_MODELS.md`;
- `PROMPTS_SPEC.md`;
- `API_SPEC.md`;
- `ADMIN_PANEL_SPEC.md`.

Документ не является формальным QA-планом с оценками трудозатрат и не задаёт инструмент автоматизации. Он задаёт список обязательных проверок, который должен быть реализован вручную и по возможности автоматизирован.

---

## 2. Подход к тестированию первой версии

В первой версии рекомендуется опираться на следующие уровни проверок:
- smoke-тесты критического пути;
- unit-тесты бизнес-логики и сервисов;
- integration-тесты пользовательских и административных потоков;
- базовые security-проверки;
- регрессионные проверки после изменения моделей, прав, промтов, LLM-параметров и логики завершения диалога.

Особое внимание должно быть уделено следующим рисковым зонам:
- один активный диалог на пользователя;
- корректное завершение диалога по кнопке, таймеру и закрытию вкладки;
- синхронный анализ и его ошибки;
- права доступа администратора, назначенного супер-администратора и главного супер-администратора;
- архивирование без разрушения исторических данных;
- корректность экспортируемого PDF;
- сохранение и отображение UI-текстов и seed-данных.

## 3. Обозначения

### 3.1. Приоритеты

- `P0` — критическая проверка, блокирующая релиз.
- `P1` — важная проверка, необходимая для основной работы системы.
- `P2` — желательная проверка, покрывающая важные, но не блокирующие сценарии.

### 3.2. Типы проверок

- `SMOKE` — быстрая проверка жизнеспособности системы.
- `UNIT` — тест уровня функции, метода, сервиса или валидатора.
- `INT` — интеграционная проверка нескольких компонентов.
- `E2E` — сквозная пользовательская проверка.
- `SEC` — проверка безопасности и ограничений доступа.

## 4. Минимальный smoke-набор перед каждым релизом

Ниже перечислен обязательный smoke-набор, который должен проходить перед каждым релизом.

### TC-SMOKE-001 — открытие сайта
- Приоритет: `P0`
- Тип: `SMOKE`
- Предусловие: приложение запущено.
- Шаги:
  1. Открыть главную страницу сайта.
- Ожидаемый результат:
  - сервер отвечает успешно;
  - отображается стартовый экран, логин или редирект по ожидаемой логике;
  - нет 500-ошибок и сломанных шаблонов.

### TC-SMOKE-002 — регистрация пользователя
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. Зарегистрировать нового пользователя с никнеймом, email и паролем.
- Ожидаемый результат:
  - пользователь создаётся;
  - входная сессия создаётся по предусмотренной логике;
  - пользователь попадает в пользовательскую часть приложения.

### TC-SMOKE-003 — вход существующего пользователя
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. Войти под существующей учётной записью.
- Ожидаемый результат:
  - вход выполняется успешно;
  - пользователь попадает на главную страницу.

### TC-SMOKE-004 — запуск сценария
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. Открыть главную страницу.
  2. Нажать кнопку одного из сценариев.
- Ожидаемый результат:
  - открывается страница чата;
  - блок `conditions` закреплён в верхней части экрана;
  - персонаж автоматически публикует стартовую реплику.

### TC-SMOKE-005 — отправка сообщения и получение ответа ИИ
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. Отправить текстовое сообщение в активном диалоге.
- Ожидаемый результат:
  - сообщение пользователя сохраняется и появляется в чате;
  - отображается индикатор ожидания ответа ИИ;
  - ответ ИИ появляется без полной перезагрузки страницы.

### TC-SMOKE-006 — завершение диалога по кнопке
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. В активном диалоге с минимум одной пользовательской репликой нажать «Дай обратную связь».
- Ожидаемый результат:
  - диалог переводится в завершённое состояние;
  - запускается анализ;
  - открывается страница результата.

### TC-SMOKE-007 — отображение результатов анализа
- Приоритет: `P0`
- Тип: `SMOKE`
- Ожидаемый результат:
  - отображается сумма вида `N из M`;
  - отображаются карточки по каждому аналитическому критерию;
  - у каждой карточки есть текст и целочисленный балл.

### TC-SMOKE-008 — экспорт в PDF
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. На странице результата нажать экспорт PDF.
- Ожидаемый результат:
  - PDF генерируется без 500-ошибки;
  - файл содержит игру, сценарий, дату, статус завершения, сумму баллов, карточки анализа и транскрипт.

### TC-SMOKE-009 — вход в админку главным супер-администратором
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. Войти под seed-учёткой `superadmin`.
  2. Открыть внутреннюю административную панель.
- Ожидаемый результат:
  - панель открывается;
  - доступны административные разделы согласно роли главного супер-администратора.

### TC-SMOKE-010 — выход из системы
- Приоритет: `P0`
- Тип: `SMOKE`
- Шаги:
  1. Нажать «Выйти».
- Ожидаемый результат:
  - сессия завершается;
  - пользователь теряет доступ к защищённым страницам.

## 5. Пользовательская часть: регистрация, вход, профиль

### TC-AUTH-001 — успешная регистрация
- Приоритет: `P0`
- Тип: `INT`
- Шаги:
  1. Заполнить валидные никнейм, email, пароль и подтверждение пароля.
  2. Отправить форму.
- Ожидаемый результат:
  - пользователь создаётся;
  - email сохраняется уникально;
  - пароль хранится в зашифрованном виде;
  - создаётся профиль и связанные стартовые данные, если это предусмотрено логикой.

### TC-AUTH-002 — запрет регистрации с уже существующим email
- Приоритет: `P0`
- Тип: `INT`
- Ожидаемый результат:
  - система не создаёт дубль;
  - показывает понятную ошибку валидации.

### TC-AUTH-003 — несовпадение пароля и подтверждения
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - форма не отправляется успешно;
  - пользователь получает понятную ошибку.

### TC-AUTH-004 — редактирование никнейма пользователем
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - никнейм обновляется;
  - аватар по первой букве обновляется, если изменилась первая буква;
  - доступ к чужим данным не появляется.

### TC-AUTH-005 — восстановление пароля по email
- Приоритет: `P0`
- Тип: `INT`
- Шаги:
  1. Запросить reset по существующему email.
  2. Использовать присланную ссылку.
  3. Установить новый пароль.
- Ожидаемый результат:
  - создаётся корректный одноразовый reset-flow;
  - пароль обновляется;
  - старый пароль перестаёт работать;
  - новый пароль позволяет войти.

### TC-AUTH-006 — вход с неверным паролем
- Приоритет: `P1`
- Тип: `SEC`
- Ожидаемый результат:
  - вход не выполняется;
  - система не раскрывает лишних деталей;
  - событие может быть отражено в логах по принятой политике.

### TC-AUTH-007 — защита закрытых страниц после logout
- Приоритет: `P0`
- Тип: `SEC`
- Ожидаемый результат:
  - после выхода защищённые страницы недоступны без повторного входа.

### TC-AUTH-008 — seed-учётные записи создаются автоматически
- Приоритет: `P0`
- Тип: `INT`
- Ожидаемый результат:
  - при первичном запуске существуют `superadmin` и `testuser`;
  - они помечены как seed-учётные записи.

### TC-AUTH-009 — супер-администратор может изменить данные seed-пользователей
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - email, никнейм и пароль seed-пользователя обновляются;
  - признак seed сохраняется;
  - изменения отражаются в аудите, если он предусмотрен логикой.

## 7. Игровой чат

### TC-CHAT-001 — автоматический старт диалога
- Приоритет: `P0`
- Тип: `E2E`
- Предусловие: выбран сценарий.
- Ожидаемый результат:
  - создаётся новый диалог в статусе active;
  - стартовая реплика персонажа публикуется автоматически;
  - приветствие берётся из `opening_message_text` сценария.

### TC-CHAT-002 — `conditions` закреплены сверху
- Приоритет: `P1`
- Тип: `E2E`
- Ожидаемый результат:
  - блок условий виден при открытии чата;
  - при прокрутке чат-ленты он остаётся закреплённым вверху.

### TC-CHAT-003 — отправка сообщения без полной перезагрузки страницы
- Приоритет: `P0`
- Тип: `E2E`
- Ожидаемый результат:
  - сообщение отправляется асинхронно;
  - страница не перезагружается полностью;
  - история чата дополняется на текущем экране.

### TC-CHAT-004 — индикатор ожидания ответа ИИ
- Приоритет: `P0`
- Тип: `INT`
- Ожидаемый результат:
  - сразу после отправки пользовательской реплики появляется индикатор «думаю...»;
  - после получения ответа индикатор исчезает.

### TC-CHAT-005 — запрет пустого сообщения
- Приоритет: `P0`
- Тип: `INT`
- Ожидаемый результат:
  - пустая строка или строка из пробелов не отправляется;
  - пользователь получает внятную ошибку или кнопка неактивна.

### TC-CHAT-006 — ограничение длины пользовательского сообщения
- Приоритет: `P0`
- Тип: `UNIT` + `INT`
- Предусловие: лимит по умолчанию 2500 символов.
- Ожидаемый результат:
  - сообщение длиннее лимита отклоняется;
  - ошибка формулируется через UI-текст/валидацию;
  - значение лимита берётся из настройки, а не зашито жёстко.

### TC-CHAT-007 — ИИ получает весь диалог
- Приоритет: `P1`
- Тип: `UNIT` + `INT`
- Ожидаемый результат:
  - в payload LLM передаётся весь накопленный транскрипт текущего диалога в правильном порядке.

### TC-CHAT-008 — один активный диалог на пользователя
- Приоритет: `P0`
- Тип: `INT`
- Шаги:
  1. Открыть один активный диалог.
  2. Попробовать открыть другой сценарий.
- Ожидаемый результат:
  - второй активный диалог не создаётся;
  - пользователь либо возвращается в текущий активный диалог, либо получает предусмотренное системой сообщение.

### TC-CHAT-009 — запрет двойной отправки одного и того же сообщения кликом
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - при повторном быстром нажатии не создаются дубли сообщений;
  - сервер не делает два LLM-запроса на одно и то же пользовательское действие.

### TC-CHAT-010 — защита от повторной отправки во время pending-запроса
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - пока ответ ИИ не получен, повторная отправка блокируется по выбранной логике;
  - состояние UI корректно отражает ожидание.

### TC-CHAT-011 — корректная работа при длинном диалоге
- Приоритет: `P2`
- Тип: `INT`
- Ожидаемый результат:
  - длинный диалог сохраняется и отображается без потери порядка сообщений;
  - интерфейс не ломается при значительном числе реплик.

### TC-CHAT-012 — отображение аватара пользователя
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - аватар строится из первой буквы никнейма;
  - фон детерминирован для пользователя;
  - буква чёрная и читаемая.

### TC-CHAT-013 — отображение сценарийного изображения, если оно задано
- Приоритет: `P2`
- Тип: `INT`
- Ожидаемый результат:
  - изображение показывается по зафиксированной UI-логике;
  - отсутствие изображения не ломает страницу.

### TC-CHAT-014 — тексты UI берутся из таблицы UI-текстов
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - изменённый через админку UI-текст отображается в пользовательском интерфейсе без хардкода.

## 8. Завершение диалога и таймер

### TC-END-001 — завершение по кнопке «Дай обратную связь»
- Приоритет: `P0`
- Тип: `E2E`
- Ожидаемый результат:
  - диалог получает статус completed;
  - фиксируется источник завершения manual;
  - чат блокируется;
  - запускается анализ.

### TC-END-002 — завершение по таймеру
- Приоритет: `P0`
- Тип: `E2E`
- Предусловие: время диалога истекло.
- Ожидаемый результат:
  - показывается сообщение «Время диалога истекло»;
  - сообщение удерживается 3 секунды;
  - пользователь переводится на экран анализа;
  - фиксируется источник завершения timeout.

### TC-END-003 — изменение глобального таймера супер-администратором
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - новое значение применяется к новым диалогам;
  - старые активные диалоги ведут себя по зафиксированной логике версии;
  - настройка сохраняется централизованно.

### TC-END-004 — закрытие вкладки после хотя бы одной реплики пользователя
- Приоритет: `P0`
- Тип: `INT`
- Шаги:
  1. Отправить хотя бы одно пользовательское сообщение.
  2. Закрыть вкладку или уйти со страницы.
- Ожидаемый результат:
  - диалог помечается как aborted / не завершён;
  - запускается анализ;
  - результаты сохраняются.

### TC-END-005 — закрытие вкладки без единой реплики пользователя
- Приоритет: `P0`
- Тип: `INT`
- Ожидаемый результат:
  - диалог не анализируется;
  - LLM-вызовы для аналитики не выполняются;
  - история отражает незавершённый/технический исход по принятой логике.

### TC-END-006 — завершение по `sendBeacon`
- Приоритет: `P1`
- Тип: `INT`
- Ожидаемый результат:
  - клиент пытается отправить сигнал завершения при уходе со страницы;
  - сервер корректно принимает сигнал и завершает диалог.

### TC-END-007 — завершение по серверному fallback-таймауту
- Приоритет: `P1`
- Тип: `INT`
- Предусловие: `sendBeacon` не был доставлен.
- Ожидаемый результат:
  - сервер завершает диалог по fallback-механизму неактивности;
  - статус и причина завершения фиксируются корректно.

## 9. Аналитика

### TC-AN-001 — запуск анализа после завершённого диалога
- Приоритет: `P0`
- Тип: `INT`

  - операция идемпотентна с точки зрения пользовательского результата.

---

## 18. Рекомендуемый минимум автоматизации

Для первой версии рекомендуется автоматизировать как минимум следующие группы проверок:
- регистрация, вход, logout и reset пароля;
- один активный диалог на пользователя;
- завершение диалога по кнопке и по таймеру;
- отказ от анализа без пользовательских реплик;
- успешный анализ с сохранением нескольких критериев;
- обработка невалидного JSON аналитики;
- генерация PDF;
- права ролей в административной панели;
- запрет администратору менять чужие сущности;
- изменение UI-текста и его отображение в пользовательском интерфейсе;
- изменение LLM-настроек супер-администратором;
- ключевые security-проверки на доступ к чужим диалогам и результатам.

## 19. Критерий готовности первой версии по тестам

Первая версия может считаться готовой к приёмке, если:
- все тесты `P0` проходят;
- нет известных критических багов в регистрации, входе, чате, завершении диалога, анализе, PDF и ролях;
- ключевые проверки `P1` по ролям, архивированию, UI-текстам и LLM-обработке пройдены;
- критические security-проверки не выявили нарушений доступа;
- система не оставляет активные диалоги в неконсистентном состоянии после штатных и аварийных сценариев.

Статус документа: согласован для первой версии.
