# USER_FLOWS.md

## 1. Назначение документа

Документ описывает пользовательские сценарии веб-тренажёра обратной связи на уровне действий пользователя, реакций системы и ожидаемых переходов между состояниями.

Документ нужен как опора для реализации:
- пользовательских экранов;
- backend-логики;
- правил переходов между страницами и статусами;
- проверок доступа;
- тестовых сценариев.

Документ описывает только согласованные сценарии первой версии и явно отмеченные заделы на будущее.

---

## 2. Участники системы

### 2.1. Гость

Неавторизованный посетитель сайта.

### 2.2. Пользователь

Зарегистрированный и авторизованный участник тренажёра.

### 2.3. Администратор

Пользователь с правом создавать и редактировать свои игры и сценарии.

### 2.4. Назначенный супер-администратор

Пользователь с расширенным доступом к глобальным настройкам, промтам, UI-текстам, всем играм и сценариям.

### 2.5. Главный супер-администратор

Пользователь с максимальными правами, включая управление администраторами и супер-администраторами.

---

## 3. Основные пользовательские состояния

### 3.1. Состояния аутентификации

- не авторизован;
- авторизован как обычный пользователь;
- авторизован как администратор;
- авторизован как назначенный супер-администратор;
- авторизован как главный супер-администратор.

### 3.2. Состояния игрового диалога

- диалог не начат;
- диалог активен;
- диалог завершён вручную;
- диалог завершён по таймеру;
- диалог прерван пользователем;
- анализ запущен;
- анализ завершён;
- анализ не запускался из-за отсутствия пользовательских реплик.

---

## 4. Глобальные правила переходов

1. Пользователь не может иметь более одного активного диалога одновременно.
2. Если у пользователя уже есть активный диалог, система не должна позволять начать новый.
3. Анализ запускается только если в диалоге была хотя бы одна пользовательская реплика.
4. Если пользователь покинул страницу диалога, диалог должен быть завершён со статусом «не завершён».
5. Если клиентский сигнал о закрытии страницы не дошёл до сервера, диалог должен быть добит сервером по таймауту неактивности.
6. Conditions всегда закреплены в верхней части страницы диалога.
7. Стартовая реплика персонажа создаётся автоматически при входе в сценарий.
8. Общая сумма баллов не хранится в БД и пересчитывается при показе результата.
9. Личный кабинет и энциклопедия в первой версии существуют как точки входа в будущие разделы, но работают как заглушки.

## 5. Пользовательские сценарии первой версии

## 5.1. Регистрация нового пользователя

### Цель

Создать учётную запись для доступа к тренажёру.

### Предусловия

- пользователь не авторизован;
- пользователь находится на публичной части сайта.

### Основной поток

1. Пользователь открывает страницу регистрации.
2. Система показывает форму регистрации.
3. Пользователь вводит:
   - никнейм;
   - email;
   - пароль.
4. Пользователь отправляет форму.
5. Система валидирует поля.
6. Система создаёт пользователя.
7. Система сообщает об успешной регистрации.
8. Пользователь входит в систему.
9. Система переводит пользователя на главную страницу приложения.

### Альтернативные потоки

#### 5.1.A. Никнейм невалиден

1. Пользователь отправляет форму с невалидным никнеймом.
2. Система показывает сообщение об ошибке у соответствующего поля.
3. Учётная запись не создаётся.

#### 5.1.B. Email уже занят

1. Пользователь отправляет форму с email, который уже существует в системе.
2. Система показывает ошибку.
3. Пользователь остаётся на форме регистрации.

#### 5.1.C. Пароль не проходит валидацию

1. Пользователь отправляет слишком слабый или некорректный пароль.
2. Система показывает ошибку.
3. Учётная запись не создаётся.

### Постусловия

- в системе создана новая учётная запись;
- пользователь может авторизоваться;
- у пользователя доступны базовые пользовательские разделы.

---

## 5.2. Вход в систему

### Цель

Получить доступ к личной сессии пользователя.

### Предусловия

- пользователь уже зарегистрирован;
- пользователь не авторизован.

### Основной поток

1. Пользователь открывает страницу входа.
2. Вводит email и пароль.
3. Отправляет форму.
4. Система проверяет данные.
5. Система создаёт авторизованную сессию.
6. Пользователь переводится на главную страницу приложения.

### Альтернативные потоки

#### 5.2.A. Неверный логин или пароль

1. Пользователь отправляет некорректные данные.
2. Система показывает дружелюбное сообщение об ошибке.
3. Пользователь остаётся на форме входа.

#### 5.2.B. Превышен допустимый темп попыток входа

1. Пользователь или бот делает слишком много попыток входа.
2. Система временно ограничивает новые попытки.
3. Пользователю показывается соответствующее сообщение.
4. Событие логируется.

### Постусловия

- пользователь авторизован;
- доступны разделы в соответствии с ролью.

---

## 5.3. Выход из системы

### Цель

Завершить авторизованную пользовательскую сессию.

### Предусловия

- пользователь авторизован.

### Основной поток

1. Пользователь нажимает кнопку «Выйти».
2. Система завершает авторизованную сессию.
3. Пользователь переводится на публичную страницу входа.

### Постусловия

- пользователь разлогинен;
- защищённые разделы больше недоступны без повторного входа.

---

## 5.4. Восстановление пароля

### Цель

Восстановить доступ к учётной записи через email.

### Предусловия

- пользователь не может войти в систему;
- email учётной записи существует в системе.

### Основной поток

1. Пользователь открывает экран восстановления пароля.
2. Вводит email.
3. Отправляет форму.
4. Система генерирует стандартный токен восстановления.
5. Система отправляет письмо с ссылкой восстановления.
6. Пользователь переходит по ссылке из письма.
7. Система открывает форму нового пароля.
8. Пользователь вводит новый пароль.
9. Система обновляет пароль.
10. Пользователь может войти с новым паролем.

### Альтернативные потоки

#### 5.4.A. Email не найден

1. Пользователь вводит email, отсутствующий в системе.
2. Система не раскрывает факт существования или отсутствия учётной записи слишком явно.
3. Пользователь получает нейтральное системное сообщение.

#### 5.4.B. Ссылка истекла или недействительна

1. Пользователь переходит по просроченной или некорректной ссылке.
2. Система сообщает, что восстановление нужно запросить заново.

### Постусловия

- пароль пользователя обновлён;
- старый пароль больше не действует.

---

## 5.5. Изменение никнейма

### Цель

Позволить пользователю изменить отображаемое имя.

### Предусловия

- пользователь авторизован.

### Основной поток

1. Пользователь открывает доступный экран редактирования профиля или соответствующую форму, когда она станет доступна.
2. Вводит новый никнейм.
3. Отправляет форму.
4. Система валидирует значение.
5. Система сохраняет новый никнейм.
6. Во всех новых пользовательских интерфейсах используется обновлённое имя.

### Примечание

В первой версии полноценный личный кабинет не реализуется. Возможность изменения никнейма должна быть предусмотрена архитектурно и может быть реализована через отдельный минимальный экран или админский путь в рамках технической реализации.

---

## 5.6. Главная страница после входа

### Цель

Дать пользователю точку входа в игру и базовую навигацию.

### Предусловия

- пользователь авторизован;
- пользователь не находится внутри активного незавершённого диалога, который блокирует запуск нового.

### Основной поток

1. Пользователь входит в систему.
2. Система открывает главную страницу.
3. На странице показываются:
   - заголовок активной игры или группы игр;
   - кнопки сценариев;
   - заглушка «Личный кабинет»;
   - заглушка «Энциклопедия»;
   - кнопка «Выйти».
4. Пользователь выбирает нужный сценарий.

### Seed-поток первой версии

Для демонстрационного наполнения после первичного запуска системы на главной странице должна быть доступна одна seed-игра с тремя кнопками сценариев. Во всех трёх сценариях используется персонаж Алексей; содержимое позже редактируется через админку.

### Альтернативные потоки

#### 5.6.A. Пользователь нажал «Личный кабинет»

1. Пользователь нажимает кнопку.
2. Система открывает страницу-заглушку.
3. Пользователь видит, что раздел будет реализован позже.

#### 5.6.B. Пользователь нажал «Энциклопедия»

1. Пользователь нажимает кнопку.
2. Система открывает страницу-заглушку.
3. Пользователь видит, что раздел будет реализован позже.

---

## 5.7. Запуск сценария

### Цель

Открыть страницу игрового диалога по выбранному сценарию.

### Предусловия

- пользователь авторизован;
- у пользователя нет другого активного диалога;
- сценарий активен и доступен пользователю.

### Основной поток

1. Пользователь нажимает кнопку выбранного сценария.
2. Система создаёт новый экземпляр диалога со статусом «активен».
3. Система фиксирует выбранную игру и сценарий.
4. Система определяет настройки таймера для пользователя:
   - в первой версии используется глобальное значение;
   - будущая персональная настройка ещё не применяется.
5. Система открывает страницу чата.
6. В верхней части страницы показывается закреплённый блок conditions.
7. В чат автоматически добавляется стартовая реплика персонажа, заданная в сценарии.
8. Пользователь видит поле ввода, кнопку отправки и кнопку «Дай обратную связь».
9. Таймер запускается.

### Альтернативные потоки

#### 5.7.A. У пользователя уже есть активный диалог

1. Пользователь пытается открыть новый сценарий.
2. Система не создаёт второй активный диалог.
3. Пользователь получает сообщение о невозможности начать новый диалог.
4. Система предлагает вернуться в текущий активный диалог или дождаться его завершения.

#### 5.7.B. Сценарий недоступен

1. Пользователь пытается открыть архивный или удалённый из показа сценарий.
2. Система возвращает контролируемую ошибку доступа или отсутствия.
3. Пользователь перенаправляется на главную страницу.

---

## 5.8. Игровой диалог: отправка сообщения пользователем

### Цель

Позволить пользователю вести ролевой диалог с ИИ-персонажем.

### Предусловия

- страница сценария открыта;
- диалог активен;
- таймер не истёк;
- пользователь не отправляет другой запрос в тот же момент.

### Основной поток

1. Пользователь вводит текстовое сообщение.
2. Нажимает отправку.
3. Система проверяет:
   - что сообщение не пустое;
   - что длина не превышает установленный лимит;
   - что диалог ещё активен;
   - что нет второго параллельного запроса на отправку.
4. Сообщение сохраняется в диалоге как реплика пользователя.
5. В чат добавляется визуальный индикатор ожидания ответа ИИ.
6. Backend формирует запрос к LLM на основе:
   - системного промта сценария;
   - всего накопленного диалога.
7. Система отправляет запрос к LLM.
8. После получения ответа индикатор ожидания заменяется репликой персонажа.
9. Чат автоматически прокручивается вниз.
10. Пользователь может продолжить разговор.

### Альтернативные потоки

#### 5.8.A. Сообщение превышает лимит

1. Пользователь отправляет слишком длинное сообщение.
2. Система не отправляет его в LLM.
3. Пользователь видит понятную ошибку.

#### 5.8.B. Двойная отправка

1. Пользователь быстро нажимает отправку несколько раз.
2. Система принимает только один запрос.
3. Повторные запросы блокируются.
4. Событие при необходимости логируется.

#### 5.8.C. Ошибка LLM при игровом ответе

1. Система не получает валидный ответ от LLM или получает техническую ошибку.
2. Система делает предусмотренное количество повторов.
3. Если повторы не помогли, система показывает пользователю дружелюбное сообщение.
4. Ошибка логируется.
5. Диалог остаётся в контролируемом состоянии.

---

## 5.9. Игровой диалог: ручное завершение

### Цель

Позволить пользователю закончить разговор и перейти к анализу.

### Предусловия

- диалог активен.

### Основной поток

1. Пользователь нажимает кнопку «Дай обратную связь».
2. Система блокирует дальнейший ввод.
3. Система переводит диалог в статус «завершён вручную».
4. Система проверяет, есть ли хотя бы одна пользовательская реплика.
5. Если пользовательская реплика есть, система запускает анализ.
6. После завершения анализа открывается страница результата.

### Альтернативный поток

#### 5.9.A. Пользователь не отправил ни одной реплики

1. Пользователь нажимает кнопку завершения, не отправив ни одного сообщения.
2. Система завершает диалог без запуска аналитики.
3. Пользователь получает контролируемый переход из сценария без страницы анализа или с техническим сообщением, что анализ не выполнялся из-за отсутствия данных.

---

## 5.10. Игровой диалог: завершение по таймеру

### Цель

Автоматически завершить сессию при истечении времени.

### Предусловия

- диалог активен;
- таймер достиг нуля.

### Основной поток

1. Таймер достигает нуля.
2. Система блокирует возможность отправлять новые сообщения.
3. На экране показывается редактируемое UI-сообщение «Время диалога истекло».
4. Сообщение показывается 3 секунды.
5. Система переводит диалог в статус «завершён по таймеру».
6. Если была хотя бы одна пользовательская реплика, запускается анализ.
7. После завершения анализа открывается страница результата.

### Альтернативный поток

#### 5.10.A. Пользовательских реплик нет

1. Таймер истекает до первой пользовательской реплики.
2. Диалог завершается по таймеру.
3. Анализ не запускается.

---

## 5.11. Игровой диалог: прерывание при закрытии вкладки или уходе со страницы

### Цель

Корректно завершить диалог, если пользователь покидает страницу до явного завершения.

### Предусловия

- диалог активен;
- пользователь закрывает вкладку, браузер или уходит на другую страницу.

### Основной поток

1. Клиент пытается отправить серверу сигнал завершения через механизм фоновой доставки при закрытии страницы.
2. Если сигнал доходит, система помечает диалог как «не завершён».
3. Если в диалоге есть хотя бы одна пользовательская реплика, запускается анализ.
4. Если реплик нет, анализ не запускается.
5. Если сигнал не дошёл, сервер завершает диалог по таймауту неактивности.
6. При серверном завершении применяются те же правила анализа.

### Постусловия

- активный диалог не остаётся висеть бесконечно;
- пользователь не может использовать один и тот же незавершённый активный диалог для обхода ограничения «один активный диалог».

---

## 5.12. Просмотр результатов анализа

### Цель

Показать пользователю структурированный результат по всем аналитическим критериям.

### Предусловия

- диалог завершён;
- анализ выполнен хотя бы по одному критерию.

### Основной поток

1. Пользователь попадает на страницу результатов.
2. В верхней части страницы система показывает:
   - название игры;
   - название сценария;
   - статус завершения;
   - сумму баллов вида «N из M».
3. Ниже показываются карточки по каждому аналитическому критерию.
4. Каждая карточка содержит:
   - название критерия;
   - балл;
   - текст анализа.
5. На странице доступна кнопка экспорта в PDF.

### Альтернативные потоки

#### 5.12.A. Часть аналитических критериев завершилась с ошибкой

1. Некоторые аналитические промты не отработали корректно.
2. Система показывает доступные результаты.
3. Пользователь получает дружелюбное системное уведомление, что часть анализа могла быть недоступна.
4. Ошибки логируются.

---

## 5.13. Экспорт результатов в PDF

### Цель

Дать пользователю экспортируемый документ с итогом сессии.

### Предусловия

- результат анализа доступен.

### Основной поток

1. Пользователь нажимает кнопку «Экспорт результатов в PDF».
2. Система формирует PDF-документ.
3. В PDF включаются:
   - название игры;
   - название сценария;
   - дата и время;
   - статус завершения;
   - сумма баллов «N из M»;
   - карточки по всем критериям;
   - полный транскрипт диалога.
4. Пользователь получает готовый PDF-файл.

### Альтернативный поток

#### 5.13.A. Ошибка генерации PDF

1. Система не смогла корректно собрать файл.
2. Пользователь получает дружелюбное сообщение об ошибке.
3. Событие логируется.

## 7. Административные сценарии

## 7.1. Вход администратора в административную часть

### Цель

Получить доступ к функциям управления продуктом.

### Предусловия

- пользователь имеет административную роль.

### Основной поток

1. Администратор входит под своей учётной записью.
2. Система определяет его роль.
3. Система открывает доступ к разрешённым административным разделам.
4. Администратор может пользоваться:
   - стандартным Django Admin;
   - внутренним административным интерфейсом продукта.

---

## 7.2. Создание игры администратором

### Цель

Создать новую игру в рамках своих прав.

### Предусловия

- пользователь имеет роль администратора, назначенного супер-администратора или главного супер-администратора.

### Основной поток

1. Администратор открывает раздел игр.
2. Нажимает «Создать игру».
3. Заполняет форму игры.
4. Система валидирует данные.
5. Система создаёт игру и связывает её с автором.
6. Игра становится доступной для дальнейшей настройки.

### Ограничения

- обычный администратор управляет только своими играми;
- супер-администраторы могут управлять всеми играми.

---

## 7.3. Создание сценария внутри игры

### Цель

Добавить новый сценарий в существующую игру.

### Предусловия

- у администратора есть право редактировать выбранную игру.

### Основной поток

1. Администратор открывает карточку игры.
2. Переходит к разделу сценариев.
3. Создаёт сценарий.
4. Заполняет ключевые поля:
   - название;
   - conditions;
   - стартовая реплика персонажа;
   - системный промт персонажа;
   - изображение, если используется.
5. Система валидирует данные.
6. Система сохраняет сценарий.
7. Сценарий становится доступен для показа пользователям в рамках игры.

---

## 7.4. Редактирование сценария

### Цель

Изменить существующий сценарий.

### Предусловия

- у администратора есть право на данный сценарий.

### Основной поток

1. Администратор открывает сценарий.
2. Изменяет поля.
3. Сохраняет изменения.
4. Система обновляет данные.
5. Новые пользователи получают обновлённую версию сценария.

### Ограничения

- обычный администратор не может редактировать чужой сценарий;
- назначенный супер-администратор и главный супер-администратор могут редактировать любой сценарий.

---

## 7.5. Архивация игры или сценария

### Цель

Убрать сущность из активного использования без физического удаления.

### Основной поток

1. Администратор с достаточными правами открывает игру или сценарий.
2. Выбирает действие архивации.
3. Система помечает сущность как архивную.
4. Сущность перестаёт быть доступной для новых пользовательских запусков.
5. Исторические данные и старые диалоги сохраняются.

### Примечание

Жёсткое удаление не является базовым пользовательским потоком первой версии и не должно использоваться как стандартный путь управления мастер-данными.

---

## 7.6. Управление аналитическими промтами

### Цель

Создавать и менять набор критериев анализа для игры.

### Предусловия

- у пользователя есть право редактировать соответствующую игру.

### Основной поток

1. Администратор открывает игру.
2. Переходит в раздел аналитических промтов.
3. Создаёт новый аналитический промт или редактирует существующий.
4. Для каждой записи задаёт:
   - название критерия;
   - alias или аналогичный технический идентификатор;
   - текст промта;
   - порядок исполнения;
   - статус активности.
5. Система сохраняет изменения.
6. Все сценарии этой игры начинают использовать обновлённый набор критериев.

### Особый seed-поток первой версии

В первой версии должны быть заведены два аналитических промта, переданных пользователем:
- «Фокус на поведении, а не на личности»;
- «Конкретность фактов».

### Задел на масштабирование

Система должна поддерживать расширение этого набора примерно до 10–12 критериев и более без изменения архитектуры.

3. Редактирует текст промта.
4. Сохраняет изменения.

## 7.9. Управление LLM-параметрами

Дать супер-администратору возможность управлять параметрами генерации.

- главный супер-администратор:
  - логин: `superadmin`;
  - пароль: `123456`;
- тестовый пользователь:
  - логин: `testuser`;

### Правила

1. Эти учётные записи создаются как технические seed-данные для первого запуска.

3. В документации и реализации должно быть отдельно отмечено, что такие пароли допустимы только как стартовые технические значения для разработки и тестирования.

## 8. Технические и аварийные пользовательские сценарии

## 8.1. Ошибка LLM в игровом диалоге

### Поток

1. Пользователь отправляет сообщение.
2. Система делает запрос в LLM.
3. Внешний сервис отвечает ошибкой или не отвечает.
4. Система выполняет внутреннюю стратегию повтора.
5. Если запрос не восстановлен, пользователю показывается дружелюбное сообщение.
6. Событие записывается в технический лог.

---

## 8.2. Ошибка LLM в аналитике

### Поток

1. Диалог завершён.
2. Система запускает очередной аналитический промт.
3. Внешний сервис возвращает ошибку или невалидный ответ.
4. Система повторяет попытку по внутренним правилам.
5. Если успех не достигнут, ошибка логируется.
6. Пользователь получает контролируемый результат с доступными частями анализа и дружелюбным уведомлением.

---

## 8.3. Невалидный JSON от аналитического промта

### Поток

1. Система получает ответ от LLM для аналитики.
2. Ответ не соответствует ожидаемой JSON-структуре.
3. Система пытается восстановить корректность ответа в пределах согласованной стратегии.
4. Если это невозможно, срабатывает повтор.
5. Если повтор не помог, ошибка логируется как отдельный технический случай.

---

## 8.4. Потеря соединения во время чата

### Поток

1. Пользователь находится в активном диалоге.
2. Во время отправки сообщения или ожидания ответа пропадает соединение.
3. Клиент показывает понятное уведомление о проблеме связи.
4. Backend сохраняет целостное состояние диалога на последнем подтверждённом шаге.
5. При уходе со страницы или длительной неактивности диалог завершается по общим правилам.

---

## 9. Заделы на будущие пользовательские потоки

Следующие потоки не входят в первую версию, но должны быть предусмотрены архитектурно:

1. Личный кабинет со списком всех прошлых сессий.
2. Повторный просмотр старых результатов анализа.
3. Пользовательские настройки таймера:
   - включение и выключение отображения таймера;
   - выбор персонального времени в диапазоне 5–20 минут.
4. Голосовые сообщения.
5. Android/APK-клиент на том же backend-ядре.
6. Публичный JSON API `/api/v1/...` для внешних клиентов.
7. Фоновый запуск аналитики вместо синхронного.

## 10. Критические продуктовые инварианты

1. Нельзя начать второй активный диалог, пока не завершён первый.
2. Нельзя потерять завершение диалога при уходе пользователя со страницы.
3. Нельзя запускать анализ пустого диалога без пользовательских реплик.
4. Нельзя позволять обычному администратору редактировать чужие игры и сценарии.
5. Нельзя позволять обычному администратору редактировать глобальные UI-тексты.
6. Нельзя позволять назначенному супер-администратору создавать новых администраторов.
7. Нельзя хранить только сумму без детализации по аналитическим критериям.
8. Нельзя терять исторические данные из-за обычного удаления мастер-данных.
9. Нельзя зависеть от Telegram-логики интерфейса; документ описывает именно веб-версию на Django.
