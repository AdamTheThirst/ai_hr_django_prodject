# PRODUCT_SPEC.md

## 1. Название продукта

Веб-тренажёр обратной связи.

## 2. Назначение продукта

Продукт представляет собой веб-приложение на Django для тренировки навыка обратной связи в формате ролевого диалога с ИИ.

Пользователь проходит сценарий внутри выбранной игры, общается с ИИ-персонажем в чат-интерфейсе, а после завершения получает структурированный анализ по набору критериев.

Проект является полной заменой Telegram-версии. При этом архитектура продукта должна допускать будущее подключение нативного Android/APK-клиента к тому же бэкенду без переписывания бизнес-логики.

## 3. Продуктовые цели

### 3.1. Основная цель

Дать пользователю безопасную и понятную среду для практики сложных разговоров и получения развёрнутой обратной связи по качеству собственной коммуникации.

### 3.2. Цели первой версии

Первая версия должна позволять:
- зарегистрироваться и войти в систему;
- выбрать игру и конкретный сценарий;
- пройти ролевой диалог с ИИ в веб-чате;
- завершить диалог вручную или по таймеру;
- получить анализ по нескольким критериям;
- экспортировать результат в PDF;
- администратору управлять играми, сценариями, промтами, UI-текстами и системными настройками через веб-интерфейсы.

### 3.3. Что не является целью первой версии

В первую версию не входят как полноценные завершённые модули:
- личный кабинет пользователя;
- энциклопедия;
- голосовые сообщения;
- отдельный публичный API для внешних клиентов;
- токен-стриминг ответа ИИ.

При этом продукт должен быть спроектирован так, чтобы эти направления можно было добавить позже без переделки ядра.

## 4. Продуктовые принципы

1. Mobile first.
2. Минималистичный интерфейс.
3. Визуальная логика чата должна быть привычной пользователю по современным LLM-чатам.
4. Минимум JavaScript: использовать только там, где без него нельзя обеспечить чат без перезагрузки, таймер и связанные динамические действия.
5. Все важные UI-тексты должны быть вынесены в редактируемые сущности.
6. Игры, сценарии и аналитика должны расширяться через админские интерфейсы, а не через ручную правку кода.
7. Один пользователь может иметь только один активный диалог одновременно.
8. Анализ должен запускаться только если пользователь успел отправить хотя бы одну реплику.

## 5. Термины и сущности продуктового уровня

### 5.1. Игра

Игра — верхний уровень логики тренировки.

Пример: «Я-высказывание».

### 5.2. Сценарий

Сценарий — конкретная ситуация внутри игры, в которой пользователь ведёт диалог.

Примеры внутри одной игры:
- Подчинённый;
- Коллега;
- Руководитель.

### 5.3. Conditions

Conditions — текст условий сценария, который показывается пользователю в закреплённом блоке в верхней части страницы диалога.

### 5.4. Стартовая реплика персонажа

Стартовая реплика — первое сообщение персонажа, которое автоматически появляется в чате при открытии сценария. Эта реплика задаётся отдельно для каждого сценария и должна редактироваться через админку.

### 5.5. Аналитический промт

Аналитический промт — отдельный критерий анализа завершённого диалога. Один аналитический промт оценивает ровно один критерий и возвращает:
- целочисленный балл;
- текст разбора.

Один набор аналитических промтов принадлежит игре, а не отдельному сценарию. Это означает, что все сценарии внутри одной игры анализируются одним и тем же набором аналитических критериев.

## 6. Пользовательские роли

### 6.1. Обычный пользователь

Может:
- зарегистрироваться;
- войти в систему;
- выйти из системы;
- восстановить пароль по email;
- изменить свой никнейм;
- выбрать игру и сценарий;
- пройти диалог;
- получить анализ;
- экспортировать результат в PDF.

Не может:
- управлять контентом проекта;
- менять системные настройки;
- открывать админские разделы.

### 6.2. Администратор

Может:
- создавать свои игры;
- создавать и редактировать свои сценарии;
- управлять связанными с ними сущностями в рамках своих прав.

Не может:
- изменять или удалять игры и сценарии, созданные другими администраторами;
- изменять глобальные UI-тексты;
- менять системные LLM-настройки;
- создавать новых администраторов.

### 6.3. Назначенный супер-администратор

Может:
- редактировать глобальные UI-тексты;
- менять системные настройки;
- управлять параметрами LLM;
- редактировать промты;
- управлять всеми играми и сценариями независимо от автора;
- управлять seed-данными пользователей, если это предусмотрено интерфейсом.

Не может:
- создавать новых администраторов.

### 6.4. Главный супер-администратор

Может всё, что может назначенный супер-администратор, и дополнительно:
- создавать администраторов;
- назначать права супер-администратора;
- отзывать права супер-администратора.

Главный супер-администратор в системе должен быть один.

### 7.1. Что входит в первую версию

1. Регистрация по никнейму, email и паролю.

3. Восстановление пароля по email.
4. Главная страница с выбором игры и запуском сценариев.
5. Страница чата.
6. Таймер диалога.
7. Ручное завершение диалога кнопкой «Дай обратную связь».
8. Автоматическое завершение по таймеру.
9. Корректная обработка незавершённого диалога при закрытии вкладки.
10. Анализ завершённого или оборванного диалога при наличии хотя бы одной пользовательской реплики.
11. Страница результатов анализа.
12. Экспорт результатов в PDF.
13. Два веб-интерфейса администрирования:
    - стандартный Django Admin;
    - отдельный внутренний административный интерфейс продукта.

15. Логирование технических событий в отдельную таблицу БД.

### 7.2. Что показывается как заглушка

На пользовательской стороне должны присутствовать как элементы интерфейса:
- «Личный кабинет»;
- «Энциклопедия».

В первой версии эти модули не реализуются полноценно и работают как заглушки.

## 8. Основной пользовательский путь

### 8.1. Регистрация и вход

1. Пользователь открывает сайт.
2. Видит формы регистрации и входа.
3. При регистрации вводит:
   - никнейм;
   - email;
   - пароль.
4. После успешной регистрации может войти в систему.
5. В системе доступна кнопка «Выйти».
6. В системе доступно полноценное восстановление пароля по email.

### 8.2. Выбор игры и сценария

1. После входа пользователь попадает на главную страницу.
2. На странице показывается игра и набор кнопок сценариев.
3. Для seed-данных первой версии должна быть подготовлена одна игра с тремя кнопками сценариев.
4. Во всех трёх seed-сценариях используется один и тот же персонаж Алексей; позже содержимое может быть изменено через админку.
5. На той же странице находятся кнопки-заглушки «Личный кабинет» и «Энциклопедия», а также кнопка «Выйти».

### 8.3. Вход в диалог

1. Пользователь нажимает кнопку сценария.
2. Открывается страница чата.
3. Conditions фиксированно показываются в верхней части страницы.
4. Персонаж автоматически начинает разговор стартовой репликой, заданной в сценарии.
5. Пользователь отвечает в чате.
6. Ответы ИИ подгружаются на текущую страницу без полной перезагрузки.
7. Во время ожидания ответа ИИ в чат вставляется индикатор «думаю...» в стиле современных LLM-чатов.

### 8.4. Завершение диалога

Диалог завершается одним из способов:
- пользователь нажал кнопку «Дай обратную связь»;
- истёк таймер;
- пользователь закрыл вкладку, закрыл браузер или покинул страницу.

### 8.5. Анализ результата

1. После завершения диалога система запускает анализ, если была хотя бы одна пользовательская реплика.
2. Если пользовательских реплик не было, анализ не выполняется.
3. После выполнения анализа пользователь попадает на страницу результатов.
4. На странице результатов показывается общая сумма вида «N из M» и карточки по каждому критерию.
5. Пользователь может экспортировать результат в PDF.

## 9. Логика игрового процесса

### 9.1. Общие правила

- У пользователя одновременно может быть только один активный диалог.
- В первой версии поддерживаются только текстовые сообщения.
- ИИ в игровом диалоге получает весь накопленный диалог, а не усечённый контекст.
- В промтах длина ответа регулируется текстом самого промта; при этом продукт должен предусматривать отдельные лимиты на длину пользовательских сообщений и длину ответов.
- Под будущие голосовые сообщения должна быть предусмотрена архитектурная заглушка.

### 9.2. Таймер

- Глобальное значение по умолчанию — 10 минут.
- Супер-администратор может менять это значение через админку.
- На экране диалога должен быть таймер.
- В будущем пользователь через личный кабинет сможет:
  - включать и выключать отображение таймера;
  - выбирать для себя длительность диалога в диапазоне от 5 до 20 минут.
- Персональная настройка таймера в первой версии не реализуется, но должна быть предусмотрена архитектурно.

### 9.3. Истечение времени

- При истечении таймера пользователю показывается сообщение «Время диалога истекло».
- Текст сообщения должен редактироваться через UI-тексты.
- Сообщение показывается 3 секунды.
- После этого пользователь автоматически переводится на страницу анализа.

### 9.4. Прерывание диалога пользователем

Если пользователь закрывает вкладку, браузер или покидает страницу, система должна:
- попытаться завершить диалог через клиентский сигнал завершения;
- при недоставке сигнала добить сессию по таймауту неактивности;
- пометить диалог как «не завершён»;
- выполнить анализ только при наличии хотя бы одной пользовательской реплики.

### 9.5. Conditions

Conditions должны быть визуально «прибиты» к верхней части экрана и оставаться постоянно доступными пользователю в процессе диалога.

## 10. Логика анализа

### 10.1. Общие правила

- Анализ строится не на одном итоговом промте, а на наборе независимых аналитических промтов.
- Каждый аналитический промт анализирует только один критерий.
- Количество аналитических промтов у игры может меняться.
- Для одной игры в будущем допускается набор из 10–12 аналитических критериев и более.
- В seed-данных первой версии должны быть заведены два аналитических промта, переданных пользователем.

### 10.2. Что сохраняется

Для каждого аналитического критерия сохраняются:
- рейтинг;
- текст анализа.

Отдельно суммарный балл по всем критериям не сохраняется. Он рассчитывается на лету при показе результата.

### 10.3. Как показывается результат

На странице результата:
- вверху показывается сумма вида «N из M»;
- ниже показываются карточки по каждому критерию;
- все карточки видны сразу на одной странице.

### 10.4. Запуск анализа

В первой версии анализ выполняется синхронно.

При этом продукт должен быть организован так, чтобы в будущем можно было перевести анализ на фоновое выполнение без изменения пользовательской логики.

### 10.5. Ошибки анализа

Если LLM:
- вернула ошибку;
- вернула невалидный JSON;
- не выполнила ожидаемый формат;

система должна:
- выполнить предусмотренную стратегию повтора;
- показать пользователю дружелюбное сообщение;
- сохранить технический лог ошибки в БД.

## 11. PDF-экспорт

Экспорт в PDF должен включать:
- название игры;
- название сценария;
- дату и время;
- статус завершения;
- общую сумму баллов в формате «N из M»;
- карточки по каждому аналитическому критерию;
- полный транскрипт диалога.

## 12. UI/UX-спецификация продуктового уровня

### 12.1. Общий стиль

- Минимализм.
- Mobile first.
- Аккуратный внешний вид на десктопе.
- Визуальный ориентир — принципы Material Design.
- Практическая реализация — Bootstrap 5 и локальный CSS.

### 12.2. Главная страница

На главной странице должны быть:
- визуальный блок с логотипом/изображением проекта;
- название игры;
- короткие описания сценариев;
- кнопки запуска сценариев;
- кнопка-заглушка «Личный кабинет»;
- кнопка-заглушка «Энциклопедия»;
- кнопка «Выйти».

### 12.3. Страница диалога

На странице диалога должны быть:
- закреплённый сверху блок conditions;
- таймер;
- чат-лента;
- сообщения персонажа и пользователя;
- поле ввода текста;
- кнопка отправки сообщения;
- кнопка «Дай обратную связь».

Страница не должна полностью перезагружаться при продолжении диалога.

### 12.4. Страница анализа

На странице анализа должны быть:
- заголовок или верхний блок с общей суммой баллов;
- карточки аналитики;
- кнопка возврата к тренажёру;
- кнопка-заглушка «Личный кабинет»;
- кнопка-заглушка «Энциклопедия».

### 12.5. Аватар пользователя

Вместо графической иконки пользователя используется первая буква никнейма на светлом фоне.

Требования:
- фон выбирается детерминированно и закрепляется за пользователем;
- фон не должен быть белым, чёрным, серым и ядовитым;
- оттенок должен оставаться светлым;
- буква всегда чёрная и читаемая.

### 12.6. Иконки в кнопках

У каждой кнопки должно быть предусмотрено место под иконку размером 24x24 px.

## 13. Контент и язык

- Язык интерфейса: русский.
- Язык игровых ответов ИИ: русский.
- Язык аналитики: русский.
- Подготовка мультиязычности в первой версии не требуется.

## 14. Seed-данные первой версии

В первой версии должны автоматически создаваться:

### 14.1. Пользователи

1. Главный супер-администратор:
- логин: `superadmin`
- пароль: `123456`

2. Тестовый пользователь:
- логин: `testuser`
- пароль: `123`

Эти учётные записи нужны для локального старта и тестирования. Они должны быть доступны для изменения через логику супер-администратора.

### 14.2. Контент

1. Одна игра.
2. Три seed-сценария внутри этой игры.
3. Во всех трёх сценариях на старте используется персонаж Алексей.
4. Два аналитических промта из переданных пользователем примеров.
5. Отдельный системный промт для генерации служебных полей аналитики.
6. Базовый набор UI-текстов.
7. Плейсхолдерные игровые промты и системные тексты там, где пользователь пока не передал финальное содержимое.

## 15. Управление контентом

Через административные интерфейсы должны редактироваться:
- игры;
- сценарии;
- conditions;
- стартовые реплики персонажей;
- игровые промты;
- аналитические промты;
- системный промт генерации служебных полей аналитики;
- UI-тексты;
- системные настройки;
- параметры LLM.

Для аналитических промтов должна быть предусмотрена возможность увеличивать или уменьшать их количество у игры без изменения кода приложения.

## 16. Системные настройки продуктового уровня

Супер-администратор должен иметь возможность управлять через админку как минимум следующими настройками:
- длительность диалога по умолчанию;
- максимальная длина пользовательского сообщения;
- параметры LLM;
- редактируемые UI-тексты.

Для параметров LLM в интерфейсе должны быть пояснения:
- что означает параметр;
- минимально допустимое значение;
- максимально допустимое значение.

## 17. Ограничения первой версии

1. Только веб-интерфейс как пользовательский канал.
2. Только текстовый ввод.
3. Только русский язык.
4. Анализ выполняется синхронно.
5. База данных первой версии — SQLite.
6. Медиа хранятся локально.
7. Личный кабинет и энциклопедия — только заглушки.

## 18. Критерии готовности первой версии на продуктовом уровне

Первая версия считается продуктово готовой, если:
- пользователь может зарегистрироваться, войти и выйти из системы;
- пользователь может запустить один из seed-сценариев;
- персонаж автоматически начинает диалог;
- чат работает без полной перезагрузки страницы;
- таймер работает;
- кнопка «Дай обратную связь» завершает сценарий;

## 19. Что должно быть предусмотрено на будущее

Продукт уже в первой версии должен быть подготовлен к следующим направлениям развития:
- личный кабинет пользователя;
- энциклопедия;
- пользовательская настройка таймера;
- голосовые сообщения;
- Android/APK-клиент на том же бэкенде;
- выделенный API `/api/v1/...`;
- возможный перенос с SQLite на PostgreSQL;
- перевод анализа на фоновые задачи.
