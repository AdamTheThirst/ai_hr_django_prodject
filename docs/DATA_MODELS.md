# DATA_MODELS.md

## 1. Назначение документа

Документ фиксирует модель данных первой версии веб-тренажёра обратной связи на Django.

Документ описывает:
- основные сущности и связи между ними;
- поля и ограничения моделей;
- обязательные состояния жизненного цикла;
- JSON-структуры, с которыми работает система;
- правила архивирования и сохранения истории.

Документ должен использоваться вместе с:
- `README.md`;
- `PRODUCT_SPEC.md`;
- `USER_FLOWS.md`;
- `FUNCTIONAL_REQUIREMENTS.md`;
- `NON_FUNCTIONAL_REQUIREMENTS.md`;
- `ARCHITECTURE.md`.

Если между кодом и этим документом возникает расхождение, источником истины считается актуальная согласованная спецификация. В первой версии при проектировании нужно придерживаться именно этой структуры данных, если иное не оговорено отдельно.

## 2. Общие принципы моделирования

### 2.1. ORM и база данных

Первая версия проекта реализуется на Django ORM и SQLite.

Все изменения схемы БД выполняются только через Django migrations.

### 2.2. Идентификаторы

Для большинства предметных сущностей рекомендуется использовать три уровня идентификации:
- внутренний первичный ключ `id` типа `BigAutoField`;
- внешний стабильный идентификатор `public_id` типа UUID;
- человекочитаемый `slug`, если сущность должна использоваться в URL, административной навигации или в стабильных ссылках.

Обязательность `slug` зависит от сущности и указана ниже.

### 2.3. Временные поля

Для всех значимых сущностей должны использоваться стандартные служебные поля:
- `created_at`;
- `updated_at`.

Для архивируемых сущностей дополнительно:
- `is_archived`;
- `archived_at`.

### 2.4. Архивирование вместо жёсткого удаления

Для мастер-данных проекта жёсткое удаление не является базовым сценарием.

Это относится как минимум к:
- играм;
- сценариям;
- игровым промтам;
- аналитическим промтам;
- системным промтам;
- UI-текстам;
- медиа-ресурсам.

Архивация нужна для того, чтобы:
- не ломать исторические диалоги;
- не ломать старые результаты анализа;
- сохранять возможность отката и повторного использования.

### 2.5. Владение объектами

Контентные сущности должны хранить информацию о создателе.

Это нужно для правила:
- обычный администратор управляет только своими объектами;
- назначенный супер-администратор управляет всеми объектами;
- главный супер-администратор управляет всеми объектами и ролями верхнего уровня.

### 2.6. Разделение активных и исторических данных

Нужно разделять:
- текущие активные настройки и контент;
- фактические данные уже завершённых сессий.

Уже начатый диалог не должен зависеть от будущих изменений глобальных настроек, поэтому эффективные параметры диалога должны сниматься в сам диалог при его создании.

## 4. Карта связей между сущностями

Базовая логика связей:

- один `User` может создать много игр, сценариев, промтов и медиа;
- одна `Game` содержит много `Scenario`;
- один `Scenario` использует один актуальный `ScenarioPrompt`;
- одна `Game` содержит много `AnalysisPrompt`;
- один `Scenario` может ссылаться на один медиа-ресурс, при этом один и тот же медиа-ресурс может переиспользоваться в нескольких сценариях;
- один `User` может иметь много `DialogSession`, но только один активный одновременно;
- один `DialogSession` содержит много `DialogMessage`;
- один `DialogSession` имеет не более одного `AnalysisRun` в первой версии;
- один `AnalysisRun` содержит много `AnalysisResult`;
- один `AnalysisResult` всегда относится к одному `AnalysisPrompt`;
- глобальные тексты и настройки живут отдельно от контентных сущностей.

Текстовая схема:

```text
User
 ├── Game
 │    ├── Scenario
 │    │    ├── ScenarioPrompt
 │    │    └── ScenarioMediaAsset (many scenarios may point to one asset)
 │    └── AnalysisPrompt
 ├── DialogSession
 │    ├── DialogMessage
 │    └── AnalysisRun
 │         └── AnalysisResult
 ├── SystemPrompt
 ├── ScenarioMediaAsset
 └── AuditLogEntry

PlatformSettings
UIText
```

## 5. Общие абстрактные поля

Для уменьшения дублирования в коде рекомендуется использовать базовые абстрактные модели.

### 5.1. `TimestampedModel` (abstract)

Общие поля:
- `created_at: DateTimeField(auto_now_add=True)`
- `updated_at: DateTimeField(auto_now=True)`

### 5.2. `UUIDPublicIdModel` (abstract)

Общие поля:
- `public_id: UUIDField(unique=True, editable=False)`

### 5.3. `ArchivableModel` (abstract)

Общие поля:
- `is_archived: BooleanField(default=False)`
- `archived_at: DateTimeField(null=True, blank=True)`

### 5.4. `OwnedModel` (abstract)

Общие поля:
- `created_by: ForeignKey(User, null=True, blank=True, on_delete=SET_NULL)`

Для исторических данных удаление создателя не должно разрушать записи; поэтому в большинстве случаев для `created_by` должен использоваться `SET_NULL`, а не `CASCADE`.

## 6. Модель пользователей и ролей

## 6.1. `User`

Рекомендуется использовать кастомную модель пользователя Django с `email` как основным логином.

Основание:
- вход выполняется по email и паролю;
- никнейм является отдельным изменяемым полем;
- проекту нужны дополнительные поля ролей, аватара и служебных признаков.

### Поля

| Поле | Тип | Обязательность | Ограничения / смысл |
|---|---|---:|---|
| `id` | `BigAutoField` | да | внутренний PK |
| `public_id` | `UUIDField` | да | внешний стабильный ID |
| `email` | `EmailField` | да | уникальный, используется для входа и reset password |
| `nickname` | `CharField` | да | отображаемое имя пользователя |
| `password` | стандарт Django | да | хранится только в хэшированном виде |
| `role` | `CharField(choices)` | да | одна из ролей доступа |
| `is_primary_superadmin` | `BooleanField` | да | признак главного супер-администратора |
| `avatar_letter` | `CharField(max_length=1)` | да | первая буква никнейма, вычисляемое/снимаемое поле |
| `avatar_bg_hex` | `CharField(max_length=7)` | да | детерминированный светлый цвет фона |
| `is_active` | `BooleanField` | да | стандартный флаг Django |
| `is_staff` | `BooleanField` | да | доступ в Django Admin |
| `date_joined` | `DateTimeField` | да | дата регистрации |
| `last_login` | стандарт Django | нет | дата последнего входа |
| `created_by` | `ForeignKey(User)` | нет | кто создал административную учётку, для саморегистрации может быть `NULL` |
| `created_at` | `DateTimeField` | да | служебное поле |
| `updated_at` | `DateTimeField` | да | служебное поле |

### Роли

`role` должен принимать только один из вариантов:
- `user`
- `admin`
- `superadmin`

Главный супер-администратор определяется не отдельным значением `role`, а сочетанием:
- `role = superadmin`
- `is_primary_superadmin = True`

Так проще сохранить иерархию:
- пользователь;
- администратор;
- назначенный супер-администратор;
- главный супер-администратор.

### Ограничения

1. `email` должен быть уникальным.
2. Главный супер-администратор в системе должен быть только один.
3. Если `is_primary_superadmin = True`, тогда `role` обязан быть `superadmin`.
4. Для административных учёток `is_staff` должен быть `True`.
5. Для обычного пользователя `is_staff` должен быть `False`, если не принято отдельное техническое решение.

### Комментарии по никнейму

Никнейм в первой версии является обязательным отображаемым именем, но его уникальность не требуется спецификацией. Если позже будет нужна уникальность, это должно вводиться отдельным миграционным решением.

### Комментарии по аватару

Поля `avatar_letter` и `avatar_bg_hex` нужны для стабильного UI:
- `avatar_letter` вычисляется как первая буква строки `nickname` после нормализации;
- `avatar_bg_hex` должен генерироваться детерминированно из данных пользователя;
- цвет должен принадлежать заранее утверждённой палитре светлых, неядовитых цветов, на которых чёрная буква читается безусловно.

### Seed-учётки первой версии

В dev-режиме должны быть подготовлены стартовые записи:
- `superadmin / 123456`
- `testuser / 123`

При этом:
- логин выполняется по email, поэтому для seed-данных должны быть заданы и email-адреса;
- значения паролей допустимы только как стартовые технические данные для разработки и первичной настройки;
- главный супер-администратор должен иметь возможность позже изменить эти данные через административный интерфейс.

## 7.1. `Game`

Игра — верхний контейнер сценариев и аналитических критериев.

### Поля

| Поле | Тип | Обязательность | Ограничения / смысл |
|---|---|---:|---|
| `id` | `BigAutoField` | да | PK |
| `public_id` | `UUIDField` | да | внешний стабильный ID |

| `title` | `CharField` | да | название игры |
| `short_description` | `TextField` | нет | краткое описание на главной странице или в админке |
| `sort_order` | `PositiveIntegerField` | да | порядок показа |
| `is_published` | `BooleanField` | да | доступна ли игра обычным пользователям |
| `is_archived` | `BooleanField` | да | архивность |

2. Архивная игра не должна появляться на пользовательской главной странице.

## 7.2. `Scenario`

Сценарий — конкретная ситуация внутри игры.

### Поля

| Поле | Тип | Обязательность | Ограничения / смысл |
|---|---|---:|---|
| `id` | `BigAutoField` | да | PK |
| `public_id` | `UUIDField` | да | внешний стабильный ID |
| `game` | `ForeignKey(Game)` | да | игра-владелец |
| `slug` | `SlugField` | да | уникален в пределах игры |
| `title` | `CharField` | да | название кнопки / сценария |
| `short_description` | `TextField` | нет | краткое пояснение для пользователя или админки |
| `conditions_text` | `TextField` | да | текст блока `conditions` |
| `opening_message_text` | `TextField` | да | первая реплика персонажа, автоматически попадающая в чат |
| `media_asset` | `ForeignKey(ScenarioMediaAsset)` | нет | связанное изображение/медиа |
| `sort_order` | `PositiveIntegerField` | да | порядок показа кнопок в игре |
| `is_published` | `BooleanField` | да | доступен ли сценарий пользователю |
| `is_archived` | `BooleanField` | да | архивность |
| `archived_at` | `DateTimeField` | нет | дата архивирования |
| `created_by` | `ForeignKey(User)` | нет | автор сценария |
| `created_at` | `DateTimeField` | да | служебное поле |
| `updated_at` | `DateTimeField` | да | служебное поле |

### Ограничения

1. `(game, slug)` должны быть уникальными.
2. Сценарий не может существовать без игры.
3. Сценарий должен иметь `conditions_text` и `opening_message_text`.
4. Архивный сценарий не должен быть доступен для запуска пользователем.
5. Один сценарий использует один актуальный игровой промт через отдельную модель `ScenarioPrompt`.

### Комментарии

В первой версии seed-игра должна содержать три сценария-кнопки. Во всех трёх seed-сценариях персонаж — Алексей; содержимое позже редактируется через админку.

## 7.3. `ScenarioMediaAsset`

Медиа-ресурс сценария. В первой версии это в первую очередь изображение, хранящееся локально в `MEDIA_ROOT`.

Модель должна позволять:
- переиспользовать один и тот же файл в нескольких сценариях;
- сохранять старые версии;
- архивировать устаревшие файлы без обязательного физического удаления.

### Поля

| Поле | Тип | Обязательность | Ограничения / смысл |
|---|---|---:|---|
| `id` | `BigAutoField` | да | PK |
| `public_id` | `UUIDField` | да | внешний стабильный ID |
| `title` | `CharField` | да | человекочитаемое название ресурса |
| `file` | `FileField/ImageField` | да | файл в `MEDIA_ROOT` |
| `original_filename` | `CharField` | нет | исходное имя загруженного файла |
| `mime_type` | `CharField` | нет | MIME-тип файла |
| `file_size_bytes` | `BigIntegerField` | нет | размер файла |
| `checksum_sha256` | `CharField` | нет | контрольная сумма для дедупликации и диагностики |
| `previous_version` | `ForeignKey(self)` | нет | ссылка на предыдущую версию того же ресурса |
| `uploaded_by` | `ForeignKey(User)` | нет | кто загрузил файл |
| `is_archived` | `BooleanField` | да | архивность |
| `archived_at` | `DateTimeField` | нет | дата архивирования |
| `created_at` | `DateTimeField` | да | служебное поле |
| `updated_at` | `DateTimeField` | да | служебное поле |

### Ограничения

1. Архивный медиа-ресурс не должен автоматически подставляться в новые сценарии, если не выбрано иное административно.
2. Удаление сценария не должно удалять файл, если он используется где-либо ещё.
3. Замена изображения в сценарии должна создавать новую запись `ScenarioMediaAsset` или переводить сценарий на уже существующую запись, а не перетирать исторические данные в уже использованном диалоге.

### Комментарий по версии файла

Связь `previous_version` достаточно для первой версии, чтобы хранить историю замен без отдельной сложной системы версионирования.

## 7.4. `ScenarioPrompt`

Игровой промт сценария.

Используется для ролевого поведения ИИ в игровом диалоге.

Почему отдельная модель, а не просто текстовое поле в `Scenario`:
- нужен архив;
- нужен создатель;
- нужно безопасно хранить историю изменений;
- в будущем может понадобиться несколько версий на сценарий.

### Поля

| Поле | Тип | Обязательность | Ограничения / смысл |
|---|---|---:|---|
| `id` | `BigAutoField` | да | PK |
| `public_id` | `UUIDField` | да | внешний стабильный ID |
| `scenario` | `ForeignKey(Scenario)` | да | сценарий-владелец |
| `title` | `CharField` | да | служебное название версии промта |
| `prompt_text` | `TextField` | да | полный текст игрового промта |
| `is_active` | `BooleanField` | да | активная версия промта для сценария |
| `is_archived` | `BooleanField` | да | архивность |
| `archived_at` | `DateTimeField` | нет | дата архивирования |
| `created_by` | `ForeignKey(User)` | нет | автор |
| `created_at` | `DateTimeField` | да | служебное поле |
| `updated_at` | `DateTimeField` | да | служебное поле |

### Ограничения

1. Для одного сценария в первой версии должна существовать не более одной активной записи `ScenarioPrompt`.
2. При создании новой активной версии предыдущая активная должна деактивироваться или архивироваться.
3. Диалог должен использовать ту версию игрового промта, которая была активна на момент старта диалога.

### Дополнение про сохранение истории

В уже начатом диалоге нельзя подменять промт задним числом. Для этого в `DialogSession` должен фиксироваться `scenario_prompt_used`.

## 7.5. `AnalysisPrompt`

Аналитический промт, оценивающий один критерий. Набор аналитических промтов принадлежит игре и применяется ко всем её сценариям.

### Поля

| Поле | Тип | Обязательность | Ограничения / смысл |
|---|---|---:|---|
| `id` | `BigAutoField` | да | PK |
| `public_id` | `UUIDField` | да | внешний стабильный ID |
| `game` | `ForeignKey(Game)` | да | игра-владелец |
| `alias` | `SlugField` | да | технический стабильный идентификатор критерия |
| `title` | `CharField` | да | человекочитаемое название критерия |
| `header_text` | `CharField` | да | заголовок карточки результата |
| `comment_text` | `TextField` | нет | служебный комментарий/подсказка для админки |
| `prompt_text` | `TextField` | да | полный текст аналитического промта |
| `sort_order` | `PositiveIntegerField` | да | порядок запуска и отображения |
| `min_rating` | `SmallIntegerField` | да | минимальный балл шкалы, по умолчанию 0 |
| `max_rating` | `SmallIntegerField` | да | максимальный балл шкалы, по умолчанию 5 |
| `is_active` | `BooleanField` | да | участвует ли в анализе |
| `is_archived` | `BooleanField` | да | архивность |
| `archived_at` | `DateTimeField` | нет | дата архивирования |
| `created_by` | `ForeignKey(User)` | нет | автор |
| `created_at` | `DateTimeField` | да | служебное поле |
| `updated_at` | `DateTimeField` | да | служебное поле |

### Ограничения

1. `(game, alias)` должны быть уникальными.
2. `(game, sort_order)` желательно держать уникальными, чтобы порядок критериев был однозначным.
3. `min_rating <= max_rating` обязательно.

### Комментарий

В первой версии используются два аналитических промта из исходных материалов пользователя, но модель обязана масштабироваться на набор из 10–12 и более критериев.

## 12. JSON-структуры первой версии

В первой версии проект работает в основном с реляционными таблицами, однако несколько структур должны быть зафиксированы как JSON-контракты.

## 12.1. JSON-ответ аналитической LLM

Ожидаемый минимальный контракт:

```json
{
  "rating": 0,
  "text": "..."
}
```

### Ограничения

- `rating` — целое число;
- `text` — непустая строка;
- JSON должен быть объектом верхнего уровня;
- лишние поля допустимы только если сервис решил их не считать ошибкой; в первой версии рекомендуется ориентироваться на минимальный контракт и игнорировать несогласованные расширения;
- при невалидном JSON или неправильных типах должен срабатывать retry и логирование.

### Нормализованная внутренняя схема после разбора

```json
{
  "rating": 4,
  "text": "Развёрнутый текст анализа по одному критерию.",
  "meta": {
    "validation_status": "valid",
    "attempt_count": 1
  }
}
```

Поле `meta` — внутренняя сервисная надстройка, не требуемая от LLM, но полезная в коде.

---

## 12.2. JSON для `AnalysisResult.parsed_json_snapshot`

Если результат признан валидным, рекомендуется сохранять исходную нормализованную структуру в JSONField:

```json
{
  "rating": 4,
  "text": "Развёрнутый текст анализа по одному критерию."
}
```

Это нужно для:
- трассировки разбора;
- диагностики;
- возможного будущего API.

## 13. Состояния и жизненные циклы

## 13.1. Состояния пользователя

На уровне модели `User` нет отдельной сложной state machine; используются:
- `is_active`;
- `role`;
- `is_primary_superadmin`.

Логически учётка может быть:
- обычной пользовательской;
- административной;
- супер-администраторской;
- главным супер-администратором;
- деактивированной.

---

## 13.2. Состояния игры и сценария

Для `Game`, `Scenario`, `ScenarioPrompt`, `AnalysisPrompt`, `SystemPrompt`, `ScenarioMediaAsset`, `UIText` используются два независимых измерения:
- активность/публикация;
- архивность.

### Логика

- опубликовано и не архивировано — доступно по назначению;
- не опубликовано и не архивировано — черновой/скрытый объект;
- архивировано — исторический объект, не используется по умолчанию.

Для промтов вместо `is_published` используется `is_active`.

---

## 13.3. Жизненный цикл диалога

Рекомендуемая схема:

```text
active
 ├── finished (manual_feedback)
 ├── finished (timeout)
 ├── aborted (page_leave)
 ├── aborted (inactive_timeout)
 └── analysis_skipped (no_user_messages)
```

### Правила

1. При создании диалога статус всегда `active`.
2. После нажатия «Дай обратную связь»:
   - если есть пользовательские реплики, статус переходит в `finished`, причина — `manual_feedback`;
   - если реплик нет, статус переходит в `analysis_skipped`, причина — `no_user_messages`.
3. По истечении таймера:
   - если есть реплики, статус — `finished`, причина — `timeout`;
   - если реплик нет, допустимо переводить в `analysis_skipped`, причина — `no_user_messages`.
4. При покидании страницы:
   - если клиентский сигнал дошёл, причина — `page_leave`;
   - если сигнал не дошёл и сработала серверная защита, причина — `inactive_timeout`.

---

## 13.4. Жизненный цикл анализа

Рекомендуемая схема:

```text
pending -> running -> completed
                └-> failed
```

Если анализ не должен выполняться из-за отсутствия пользовательских реплик, в первой версии предпочтительно не создавать `AnalysisRun` вообще. Если потребуется единообразие отчётности, допустим вариант `skipped`, но он не является обязательным.

---

## 13.5. Жизненный цикл результата по критерию

Для `AnalysisResult.validation_status`:

```text
valid
invalid_json
invalid_schema
fallback_saved
```

### Логика

- `valid` — JSON валиден и сохранён как штатный результат;
- `invalid_json` — модель вернула несинтаксический JSON;
- `invalid_schema` — JSON синтаксически валиден, но не соответствует контракту;
- `fallback_saved` — после исчерпания попыток сохранён технический запасной результат по правилам сервиса.

## 14. Ограничения целостности и индексы

Ниже перечислены ключевые ограничения, которые нужно по возможности поддержать на уровне БД и приложения.

### 14.1. Уникальность

Обязательно:
- `User.email` — уникален;
- `Game.slug` — уникален;
- `(Scenario.game, Scenario.slug)` — уникально;
- `(AnalysisPrompt.game, AnalysisPrompt.alias)` — уникально;
- `(DialogMessage.dialog, DialogMessage.sequence_no)` — уникально;
- `(AnalysisRun.dialog)` — уникально в первой версии;
- `(AnalysisResult.analysis_run, AnalysisResult.analysis_prompt)` — уникально;
- `UIText.key` — уникален;
- `SystemPrompt.key` — уникален по активной записи или глобально, в зависимости от выбранной реализации архива.

### 14.2. Один главный супер-администратор

Нужно обеспечить, чтобы в системе был только один пользователь с:
- `role = superadmin`
- `is_primary_superadmin = True`

Это может обеспечиваться:
- `UniqueConstraint` с условием, если целевая версия SQLite/Django это поддерживает;
- плюс обязательной проверкой на уровне сервиса и административных форм.

### 14.3. Один активный диалог на пользователя

Нужно обеспечить, чтобы у пользователя была только одна `DialogSession` со статусом `active`.

Рекомендуется:
- условное ограничение уникальности, если доступно;
- плюс обязательная сервисная проверка перед стартом новой сессии.

### 14.4. Индексы

Рекомендуемые индексы:
- `DialogSession(user, status)`
- `DialogSession(game)`
- `DialogSession(scenario)`
- `DialogSession(started_at)`
- `DialogMessage(dialog, sequence_no)`
- `AnalysisPrompt(game, sort_order, is_active)`
- `AnalysisResult(analysis_run, sort_order_snapshot)`
- `AuditLogEntry(level, created_at)`
- `AuditLogEntry(event_type, created_at)`

## 15. Политика удаления и `on_delete`

Рекомендуемые стратегии:

### `CASCADE`

Допустимо для:
- `DialogMessage -> DialogSession`
- `AnalysisResult -> AnalysisRun`

То есть удаление родителя удаляет только внутренние дочерние технические записи той же исторической сущности.

### `PROTECT`

Предпочтительно для:
- `DialogSession -> Game`
- `DialogSession -> Scenario`
- `DialogSession -> ScenarioPrompt`
- `AnalysisResult -> AnalysisPrompt`

Смысл: исторические данные не должны разрушаться из-за удаления мастер-объекта.

### `SET_NULL`

Предпочтительно для:
- `created_by`
- `uploaded_by`
- `updated_by`
- `actor_user`

Смысл: история остаётся, даже если учётка деактивирована или удалена по отдельной административной процедуре.

### Обязательно

## 17. Что не должно храниться как отдельные модели первой версии

В первой версии не требуется выделять в отдельные модели:
- сумму баллов по анализу — она считается на лету;
- визуальный индикатор «думаю...» — это UI-состояние, а не данные БД;
- пользовательские настройки таймера — они пока не активны продуктово;
- историю входов пользователя как отдельную предметную сущность, если хватает стандартных средств Django и технического логирования;
- markdown-структуры сообщений — ответы хранятся как обычный текст.

## 18. Итоговое решение по модели данных

Для первой версии достаточен и согласован следующий каркас:
- кастомный `User` с email-логином, никнеймом и иерархией ролей;

- `ScenarioPrompt` как отдельная архивируемая версия игрового промта;

- `SystemPrompt` для внутренних служебных задач;

- `DialogSession` и `DialogMessage` для игрового цикла;
- `AnalysisRun` и `AnalysisResult` для аналитики;
- `PlatformSettings` и `UIText` для глобального управления системой;

Эта структура покрывает первую веб-версию, не противоречит уже утверждённым требованиям и оставляет прямой путь к будущему подключению APK-клиента, личного кабинета и дополнительных режимов.
