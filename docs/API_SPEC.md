# API_SPEC.md

## 1. Назначение документа

Этот документ фиксирует контракты между компонентами первой версии проекта.

Под «API» в рамках первой версии понимаются не только внешние HTTP-endpoint-ы, но и внутренние сервисные контракты между слоями приложения:
- браузерный UI ↔ Django web layer;
- Django web layer ↔ application services;
- application services ↔ интеграции с LLM, email и PDF;
- application services ↔ будущий внешний `/api/v1/...` для APK.

Документ не дублирует детальное описание ORM-моделей, UI-экранов и ролей. Эти аспекты уже зафиксированы в:
- `DATA_MODELS.md`;
- `ARCHITECTURE.md`;
- `FUNCTIONAL_REQUIREMENTS.md`;
- `USER_FLOWS.md`.

---

## 2. Область действия

В первой версии проекта действуют следующие правила:
- основной пользовательский интерфейс — серверный HTML на Django templates;
- чат работает на той же странице без полной перезагрузки через минимальные асинхронные JSON-запросы;
- полноценный внешний публичный REST API для APK в первой версии не публикуется;
- архитектура должна позволять позже добавить `/api/v1/...` без переписывания бизнес-логики.

Следовательно, в первой версии обязательны:
1. HTML-контракты пользовательских и административных страниц;
2. компактные JSON-контракты для действий внутри страницы чата;
3. стабильные внутренние сервисные интерфейсы.

---

## 3. Слои контрактов

## 3.1. Веб-страницы (HTML)

Используются для:
- регистрации;
- входа;
- выхода;
- восстановления пароля;
- выбора игр и сценариев;
- просмотра страницы чата;
- просмотра страницы результатов;
- административных интерфейсов.

Типовой ответ:
- `200 OK` с HTML-страницей;
- `302/303` redirect после успешного POST;
- `200 OK` с HTML-формой и ошибками валидации при неуспешном POST;
- контролируемые `403/404` для запретных и несуществующих объектов.

## 3.2. Асинхронные действия внутри страницы чата (JSON)

Используются для:
- отправки сообщения пользователем;
- завершения диалога вручную;
- завершения по таймеру;
- сигнала о покидании страницы.

Типовой ответ:
- `application/json`;
- компактная структура без HTML-фрагментов;
- предсказуемые поля состояния диалога и сообщений.

## 3.3. Внутренние сервисные контракты

Сервисный слой должен быть основным носителем бизнес-логики.

View-функции, class-based views, admin views и будущие API views не должны напрямую реализовывать:
- управление жизненным циклом диалога;
- вызовы LLM;
- парсинг и валидацию JSON аналитики;
- расчёт итоговых баллов;
- генерацию PDF;
- выбор активных промтов и активных настроек платформы.

Все такие действия должны идти через сервисные интерфейсы, описанные ниже.

## 4. Общие правила для всех контрактов

## 4.1. Аутентификация

### Веб-слой первой версии

Используется стандартная аутентификация Django по сессии:
- login/password;
- cookie-based session;
- CSRF protection для всех state-changing запросов.

### Асинхронные JSON-запросы

Асинхронные запросы страницы чата выполняются в рамках той же пользовательской сессии и также обязаны проходить через CSRF-защиту, кроме специального сценария `sendBeacon` на покидание страницы, где допускается отдельный контролируемый механизм проверки запроса.

## 4.2. Авторизация

Все пользовательские runtime-endpoint-ы для диалогов должны проверять:
- что пользователь аутентифицирован;
- что диалог принадлежит именно этому пользователю;
- что состояние диалога допускает запрошенное действие.

Административные endpoint-ы обязаны дополнительно проверять роль и правила владения объектами.

## 4.3. Идентификаторы

В контрактах используются:
- `slug` — для человекочитаемых URL игры и сценария;
- `public_id` типа UUID — для внешней ссылки на диалог и анализ;
- внутренний numeric `id` не должен использоваться как внешний обязательный идентификатор в пользовательских URL и JSON runtime-контрактах.

## 4.4. Форматы времени

Во внутренних JSON-структурах и в будущих API-контрактах время должно передаваться в ISO 8601.

Для HTML-представления формат времени выбирается view-model слоем и может быть локализован отдельно.

## 4.5. Язык

В первой версии:
- интерфейс русский;
- служебные сообщения пользователю — русский;
- игровые и аналитические тексты — русский.

Технические `code` в JSON-ошибках допускаются в латинице.

## 4.6. Контент-форматы

### Игровой ответ

Игровой ответ LLM — обычный текст.

В первой версии не требуется:
- markdown как обязательный формат;
- HTML от модели;
- потоковая токеновая выдача.

### Аналитический ответ

Аналитический LLM-ответ должен быть JSON с минимальной схемой:

```json
{
  "rating": 0,
  "text": "..."
}
```

Это обязательный контракт для аналитического сервиса.

## 4.7. Стандартный формат JSON-ответа для асинхронных действий

Для runtime JSON-endpoint-ов первой версии должен использоваться единый конверт:

```json
{
  "ok": true,
  "code": "ok",
  "message": "",
  "data": {}
}
```

При ошибке:

```json
{
  "ok": false,
  "code": "message_too_long",
  "message": "Сообщение превышает допустимую длину.",
  "data": {}
}
```

Правила:
- `ok` — булево значение;
- `code` — машинно-читаемый код результата;
- `message` — короткое пользовательское или технико-пользовательское описание;
- `data` — полезная нагрузка, зависящая от endpoint-а.

## 5. Веб-контракты первой версии

## 5.1. Публичные HTML-маршруты

| Маршрут | Методы | Назначение | Тип ответа |
|---|---|---|---|
| `/` | `GET` | корневой вход, redirect на актуальную точку входа | redirect/HTML |
| `/auth/register/` | `GET`, `POST` | регистрация пользователя | HTML |
| `/auth/login/` | `GET`, `POST` | вход пользователя | HTML |
| `/auth/logout/` | `POST` | выход пользователя | redirect |
| `/auth/password-reset/` | `GET`, `POST` | запрос сброса пароля | HTML |
| `/games/` | `GET` | список игр | HTML |
| `/games/<game_slug>/` | `GET` | страница конкретной игры со списком сценариев | HTML |
| `/scenarios/<scenario_slug>/play/` | `GET` | старт сценария или переход в уже существующий активный диалог | HTML/redirect |
| `/dialogs/<dialog_public_id>/` | `GET` | страница чата | HTML |
| `/dialogs/<dialog_public_id>/results/` | `GET` | страница результатов | HTML |
| `/dialogs/<dialog_public_id>/export-pdf/` | `GET` | экспорт результатов в PDF | `application/pdf` |
| `/profile/` | `GET` | заглушка личного кабинета | HTML |
| `/knowledge-base/` | `GET` | заглушка энциклопедии | HTML |

## 5.2. Регистрация

### `GET /auth/register/`

Возвращает HTML-форму регистрации.

### `POST /auth/register/`

Принимает поля:
- `nickname`;
- `email`;
- `password1`;
- `password2`.

Правила:
- email должен быть уникальным;
- nickname обязателен;
- пароль валидируется стандартной политикой Django и внутренними ограничениями проекта;
- при успехе создаётся пользователь и его профиль;
- пользователь может быть автоматически залогинен после успешной регистрации.

Успешный результат:
- redirect на `/games/` или другой основной пользовательский экран.

Неуспешный результат:
- возврат той же формы с ошибками.

## 5.3. Вход

### `GET /auth/login/`

Возвращает HTML-форму входа.

### `POST /auth/login/`

Принимает:
- `email` или выбранный проектом login identifier;
- `password`.

Поскольку по продуктовой логике регистрация строится на email, в первой версии вход должен поддерживать email как основной идентификатор.

Успешный результат:
- создание пользовательской сессии;
- redirect на `/games/` или ранее запрошенный допустимый URL.

Неуспешный результат:
- возврат формы с сообщением об ошибке.

## 5.4. Выход

### `POST /auth/logout/`

Требует валидную сессию и CSRF.

Результат:
- завершение текущей сессии;
- redirect на страницу входа или другой публичный экран.

## 5.5. Сброс пароля

В первой версии должен использоваться полноценный стандартный flow Django password reset, адаптированный под проект.

Контракт включает:
- форму запроса письма на сброс;
- генерацию одноразовой ссылки;
- подтверждение нового пароля;
- серверную проверку токена.

Точный набор служебных URL может следовать стандартному Django flow и не должен ломать общее правило сессий и ролей.

## 5.6. Страница списка игр

### `GET /games/`

Возвращает список доступных пользователю игр.

Для каждой игры в view-model должны быть доступны:
- `title`;
- `slug`;
- краткое описание при наличии;
- preview-список сценариев или ссылка на страницу игры;
- флаг доступности.

Также на странице должны быть доступны:
- переход к заглушке личного кабинета;
- переход к заглушке энциклопедии;
- кнопка «Выйти».

## 5.7. Страница игры

### `GET /games/<game_slug>/`

Возвращает страницу одной игры.

View-model должна содержать:
- метаданные игры;
- список активных сценариев этой игры в порядке показа;
- для каждого сценария:
  - `title`;
  - `slug`;
  - ссылка на запуск;
  - место под иконку 24x24;
  - наличие/отсутствие изображения.

Для seed-содержимого первой версии допускается одна игра с тремя кнопками сценариев и одним персонажем Алексеем в каждом сценарии.

## 5.8. Старт сценария

### `GET /scenarios/<scenario_slug>/play/`

Назначение:
- проверить доступность сценария;
- проверить отсутствие второго активного диалога;
- создать новый диалог либо вернуть пользователя в текущий активный диалог;
- автоматически создать первое `assistant`-сообщение из `opening_message_snapshot_text`;
- открыть страницу чата.

Если у пользователя уже есть активный диалог:
- второй активный диалог не создаётся;
- пользователь получает контролируемое сообщение;
- пользователь должен иметь возможность вернуться в текущий активный диалог.

## 5.9. Страница чата

### `GET /dialogs/<dialog_public_id>/`

Возвращает HTML страницы чата.

View-model должна содержать:
- идентификатор диалога `public_id`;
- игру и сценарий;
- закреплённый `conditions_snapshot_text`;
- массив уже сохранённых сообщений диалога;
- статус диалога;
- число секунд до истечения таймера;
- флаг допустимости отправки нового сообщения;
- флаг допустимости ручного завершения;
- редактируемые UI-тексты, нужные странице;
- сведения об аватаре пользователя;
- сведения об аватаре персонажа при наличии.

Если диалог уже завершён:
- runtime-элементы отправки должны быть недоступны;
- пользователь должен быть направлен к результату или к контролируемому финальному состоянию.

## 5.10. Страница результатов

### `GET /dialogs/<dialog_public_id>/results/`

Назначение:
- показать итог диалога;
- показать сумму `N из M`;
- показать карточки по каждому сохранённому критерию;
- показать статус завершения диалога;
- дать экспорт в PDF.

Предусловия:
- диалог завершён;
- анализ либо выполнен, либо официально пропущен из-за отсутствия пользовательских реплик.

Если анализ не выполнялся из-за отсутствия пользовательских реплик:
- пользователь получает контролируемое состояние без полноформатной аналитической страницы или с техническим сообщением об отсутствии данных.

## 5.11. Экспорт в PDF

### `GET /dialogs/<dialog_public_id>/export-pdf/`

Предусловия:
- результаты анализа доступны;
- пользователь имеет право доступа к диалогу.

Ответ:
- `application/pdf`.

PDF должен включать:
- название игры;
- название сценария;
- дату и время;
- статус завершения;
- сумму `N из M`;
- карточки по каждому критерию;
- полный транскрипт диалога.

## 5.12. Административные HTML-маршруты

В первой версии административный слой состоит из:
- стандартного Django Admin;
- отдельного backoffice-интерфейса.

Их детальные CRUD-контракты не фиксируются в этом документе и должны быть описаны в отдельной админской спецификации.

Здесь закрепляется только принцип:
- административные интерфейсы используют тот же сервисный слой, что и пользовательский runtime;
- права проверяются на сервере;
- обычный администратор не может редактировать чужие игры и сценарии;
- системные настройки и глобальные UI-тексты доступны только супер-администраторам.

## 6. Асинхронные JSON-контракты страницы чата

## 6.1. Общая модель message payload

Во всех JSON-ответах, где возвращается сообщение диалога, должна использоваться единая структура:

```json
{
  "sequence_no": 2,
  "role": "assistant",
  "text": "Текст сообщения.",
  "char_count": 16,
  "created_at": "2026-03-22T10:15:12Z"
}
```

Обязательные поля:
- `sequence_no`;
- `role`;
- `text`;
- `char_count`;
- `created_at`.

В первой версии `role` принимает только:
- `user`;
- `assistant`.

## 6.2. Общая модель dialog state payload

```json
{
  "public_id": "8d8e9d0c-...",
  "status": "active",
  "ended_reason": null,
  "user_message_count": 1,
  "assistant_message_count": 2,
  "seconds_remaining": 547,
  "can_send_message": true,
  "can_finish": true,
  "results_url": "/dialogs/8d8e9d0c-.../results/"
}
```

Правила:
- `status` должен соответствовать значениям, закреплённым в `DATA_MODELS.md`;
- `seconds_remaining` используется для клиентского таймера;
- `results_url` может быть заполнен заранее, даже если результат ещё недоступен.

## 6.3. `POST /dialogs/<dialog_public_id>/send-message/`

Назначение:
- принять одну пользовательскую реплику;
- сохранить её;
- вызвать LLM в игровом режиме;
- сохранить один ответ персонажа;
- вернуть оба сообщения и обновлённое состояние диалога.

### Request JSON

```json
{
  "text": "Сообщение пользователя"
}
```

### Успешный response JSON

```json
{
  "ok": true,
  "code": "ok",
  "message": "",
  "data": {
    "dialog": {
      "public_id": "8d8e9d0c-...",
      "status": "active",
      "ended_reason": null,
      "user_message_count": 1,
      "assistant_message_count": 2,
      "seconds_remaining": 547,
      "can_send_message": true,
      "can_finish": true,
      "results_url": "/dialogs/8d8e9d0c-.../results/"
    },
    "user_message": {
      "sequence_no": 2,
      "role": "user",
      "text": "Сообщение пользователя",
      "char_count": 22,
      "created_at": "2026-03-22T10:15:12Z"
    },
    "assistant_message": {
      "sequence_no": 3,
      "role": "assistant",
      "text": "Ответ персонажа",
      "char_count": 16,
      "created_at": "2026-03-22T10:15:16Z"
    }
  }
}
```

### Ошибки

Возможные `code`:
- `empty_message`;
- `message_too_long`;
- `dialog_not_active`;
- `dialog_not_found`;
- `forbidden`;
- `duplicate_submission_blocked`;
- `llm_error`.

### Поведение

1. Пустое сообщение отклоняется.
2. Длина сообщения проверяется по `DialogSession.effective_user_message_max_chars`.
3. Повторные параллельные отправки должны блокироваться на сервере.
4. Если LLM временно недоступна, пользователю возвращается контролируемая ошибка, а диалог остаётся в согласованном состоянии.
5. В таблицу сообщений не записывается визуальный индикатор «думаю...». Он живёт только на клиенте.

## 6.4. `POST /dialogs/<dialog_public_id>/finish/`

Назначение:
- завершить активный диалог;
- отличить ручное завершение от завершения по таймеру;
- при наличии пользовательских реплик синхронно запустить анализ;
- вернуть клиенту указание на переход к результатам или в контролируемое финальное состояние.

### Request JSON

```json
{
  "reason": "manual_feedback"
}
```

Допустимые значения `reason` для этого endpoint-а в первой версии:
- `manual_feedback`;
- `timeout`.

### Успешный response JSON при наличии анализа

```json
{
  "ok": true,
  "code": "finished",
  "message": "",
  "data": {
    "dialog": {
      "public_id": "8d8e9d0c-...",
      "status": "finished",
      "ended_reason": "manual_feedback",
      "user_message_count": 3,
      "assistant_message_count": 4,
      "seconds_remaining": 0,
      "can_send_message": false,
      "can_finish": false,
      "results_url": "/dialogs/8d8e9d0c-.../results/"
    },
    "analysis": {
      "status": "completed",
      "results_count": 2
    },
    "redirect_url": "/dialogs/8d8e9d0c-.../results/"
  }
}
```

### Успешный response JSON без запуска анализа

```json
{
  "ok": true,
  "code": "analysis_skipped_no_user_messages",
  "message": "Анализ не выполнялся, потому что пользователь не отправил ни одной реплики.",
  "data": {
    "dialog": {
      "public_id": "8d8e9d0c-...",
      "status": "analysis_skipped",
      "ended_reason": "no_user_messages",
      "user_message_count": 0,
      "assistant_message_count": 1,
      "seconds_remaining": 0,
      "can_send_message": false,
      "can_finish": false,
      "results_url": "/dialogs/8d8e9d0c-.../results/"
    },
    "analysis": {
      "status": "skipped",
      "results_count": 0
    },
    "redirect_url": "/dialogs/8d8e9d0c-.../results/"
  }
}
```

### Ошибки

Возможные `code`:
- `dialog_not_active`;
- `dialog_not_found`;
- `forbidden`;
- `analysis_failed`.

### Поведение

1. Endpoint должен быть идемпотентным на уровне бизнес-результата: повторный вызов не должен создавать второй анализ.
2. Если диалог уже завершён, сервер должен вернуть текущее финальное состояние, а не ломать страницу.
3. В первой версии анализ выполняется синхронно в рамках этого запроса.

## 6.5. `POST /dialogs/<dialog_public_id>/abandon/`

Назначение:
- принять сигнал о покидании страницы;
- завершить диалог как `aborted`, если он ещё был активен;
- запустить анализ только если в диалоге есть хотя бы одна пользовательская реплика.

Этот endpoint предназначен для вызова через фоновые браузерные механизмы, включая `sendBeacon`.

### Request payload

Форма запроса может быть минимальной. Достаточно передать:

```json
{
  "reason": "page_leave"
}
```

Дополнительно допускается служебная передача метки клиентского времени, если она нужна для логов.

### Успешный ответ

Для этого endpoint-а допустимы два равноправных варианта:
- `204 No Content`;
- компактный JSON в стандартном конверте.

Главное требование первой версии: endpoint не должен зависеть от тяжёлого клиентского сценария обработки ответа, потому что при закрытии вкладки пользователь часто не увидит ответ.

### Поведение

1. Повторные вызовы допустимы и не должны приводить к повторному анализу.
2. Если сигнал не дошёл, сервер обязан уметь завершить диалог запасным механизмом по таймауту неактивности.
3. Если пользовательских реплик нет, анализ не запускается.

## 7. Внутренние сервисные контракты

Ниже перечислены обязательные сервисные интерфейсы. Названия классов и модулей могут отличаться, но контракт их ответственности должен сохраняться.

## 7.1. `ScenarioStartService`

Назначение:
- запустить сценарий;
- не допустить второго активного диалога;
- создать `DialogSession`;
- зафиксировать snapshot-поля;
- создать первое сообщение персонажа.

### Вход

- `user`;
- `scenario_slug`.

### Выход

Нормализованный результат:
- `dialog_public_id`;
- `created_new_dialog: bool`;
- `reused_existing_dialog: bool`;
- `status_code`;
- `ui_message` при необходимости.

### Обязательные проверки

- доступность сценария;
- роль и право доступа пользователя;
- отсутствие второго активного диалога;
- наличие активного игрового промта;
- наличие `conditions_text` и `opening_message_text` в согласованном состоянии.

## 7.2. `DialogMessageService`

Назначение:
- принять пользовательское сообщение;
- провалидировать его;
- сохранить в БД;
- собрать полный транскрипт;
- вызвать LLM;
- сохранить ответ персонажа.

### Вход

- `dialog_public_id`;
- `user`;
- `text`.

### Выход

- `dialog_state`;
- `user_message_dto`;
- `assistant_message_dto`;
- `error_code`/исключение доменного уровня при неуспехе.

### Гарантии

- весь накопленный диалог передаётся в LLM;
- индикатор «думаю...» не попадает в БД;
- при неуспехе сохраняется журнал ошибки;
- согласованность счётчиков сообщений не нарушается.

## 7.3. `DialogFinishService`

Назначение:
- завершить активный диалог;
- определить итоговый `ended_reason`;
- понять, запускать ли анализ;
- вернуть финальный runtime-результат.

### Вход

- `dialog_public_id`;
- `user`;
- `reason` (`manual_feedback` или `timeout`).

### Выход

- `dialog_state`;
- `analysis_state`;
- `redirect_url`;
- признак `analysis_was_started`.

### Гарантии

- повторный вызов не должен создавать дубликаты анализа;
- анализ запускается только при наличии минимум одной пользовательской реплики;
- статус диалога и статус анализа должны быть согласованы.

## 7.4. `DialogAbandonService`

Назначение:
- обработать покидание страницы;
- завершить активный диалог как `aborted`;
- инициировать анализ по правилам первой версии.

### Вход

- `dialog_public_id`;
- `user` или проверяемый session context;
- `reason = page_leave` или `inactive_timeout`.

### Выход

- факт изменения статуса;
- признак `analysis_started`;
- результат логирования.

### Гарантии

- операция безопасна к повторам;
- запасной серверный механизм завершения по неактивности использует тот же сервисный контракт.

## 7.5. `AnalysisService`

Назначение:
- выполнить анализ завершённого диалога по всем активным аналитическим промтам игры.

### Вход

- `dialog` или `dialog_public_id`.

### Выход

- `analysis_run`;
- массив `analysis_results`;
- сводка статуса выполнения.

### Шаги, обязательные по контракту

1. Проверить, что у диалога есть хотя бы одна пользовательская реплика.
2. Получить активный набор `AnalysisPrompt` игры в порядке `sort_order`.
3. Для каждого критерия:
   - построить полный транскрипт;
   - вызвать LLM;
   - разобрать JSON;
   - проверить `rating` и `text`;
   - сохранить `AnalysisResult` со snapshot-полями.
4. Завершить `AnalysisRun` согласованным итоговым статусом.

## 7.6. `AnalysisResponseParser`

Назначение:
- валидировать и нормализовать один ответ аналитической модели.

### Вход

- `raw_response_text`;
- `rating_min`;
- `rating_max`.

### Выход

Один из вариантов:
- валидный нормализованный объект `{rating, text}`;
- доменная ошибка `invalid_json`;
- доменная ошибка `invalid_schema`;
- fallback-результат по внутреннему правилу, если такой сценарий предусмотрен конфигурацией.

### Гарантии

- парсер не должен silently принимать структурно неправильный ответ как валидный;
- причина неуспеха должна быть логируема и сохраняема в `AnalysisResult.validation_error_message` или audit-логе.

## 7.7. `PromptResolutionService`

Назначение:
- выбрать актуальные игровые, аналитические и системные промты;
- обеспечить снимок используемого состояния.

### Вход

- `scenario`;
- `game`;
- `prompt_type`;
- опционально пользователь/роль.

### Выход

- активный `ScenarioPrompt`;
- активный упорядоченный список `AnalysisPrompt`;
- активный `SystemPrompt` по alias.

### Гарантии

- архивные и неактивные записи не должны возвращаться как боевые;
- выборка должна уважать правила владения и публикации.

## 7.8. `LlmGateway`

Назначение:
- быть единственной точкой вызова внешнего OpenAI-compatible endpoint.

### Игровой вызов: вход

- `model_name`;
- `temperature`;
- `top_p`;
- `max_tokens`;
- `messages[]`.

### Игровой вызов: выход

- `reply_text`;
- `provider_request_id` при наличии;
- `raw_provider_payload` при необходимости внутреннего логирования.

### Аналитический вызов: вход

- те же параметры модели;
- один аналитический промт;
- полный транскрипт диалога.

### Аналитический вызов: выход

- `raw_text_response`.

### Гарантии

- только этот слой знает детали конкретного SDK;
- внешние исключения преобразуются в внутренние доменные ошибки интеграции;
- application services не должны зависеть от конкретного клиента SDK.

## 7.9. `ResultPresentationService`

Назначение:
- собрать page/view model для экрана результатов и PDF.

### Вход

- `dialog_public_id`;
- `viewer`.

### Выход

- игра;
- сценарий;
- статус диалога;
- массив карточек критериев;
- сумма `N`;
- максимум `M`;
- транскрипт;
- метаданные для PDF.

### Гарантии

- сумма не читается из отдельного поля БД как источник истины;
- сумма каждый раз рассчитывается по сохранённым результатам.

## 7.10. `PdfExportService`

Назначение:
- сформировать PDF по результату диалога.

### Вход

- view-model результата.

### Выход

- бинарный PDF-поток;
- служебные метаданные файла при необходимости.

### Гарантии

- PDF строится из того же результата, что и веб-страница;
- содержимое PDF не должно зависеть от неподтверждённых клиентских данных.

## 7.11. `PasswordResetService`

Первая версия может опираться на стандартные Django-сервисы, но архитектурно обязана поддерживать контракт:
- запрос сброса по email;
- выдача одноразового токена;
- отправка письма;
- подтверждение нового пароля.

Если используется стандартный Django implementation, отдельная кастомная реализация сервиса не обязательна, но внешний поведенческий контракт должен сохраняться.

## 8. Интеграционные контракты

## 8.1. Контракт игрового LLM-запроса

Игровой вызов должен строиться как chat completion c использованием:
- одного `system`-сообщения из `ScenarioPrompt.prompt_text`;
- первого `assistant`-сообщения из `opening_message_snapshot_text`;
- всех накопленных сообщений диалога.

Нормативный формат:

```json
[
  {"role": "system", "content": "<scenario prompt>"},
  {"role": "assistant", "content": "<opening message>"},
  {"role": "user", "content": "<user text>"},
  {"role": "assistant", "content": "<assistant text>"}
]
```

## 8.2. Контракт аналитического LLM-запроса

Аналитический вызов должен использовать:
- системный текст конкретного `AnalysisPrompt`;
- полный транскрипт завершённого диалога;
- модель и параметры из эффективных настроек диалога/платформы.

Ответ ожидается в виде JSON с полями:
- `rating`;
- `text`.

## 8.3. Контракт генерации метаданных аналитики

Для задачи генерации `alias`, `header_text`, `comment_text` должен использоваться отдельный `SystemPrompt`.

Контракт результата должен быть зафиксирован в соответствующем административном сервисе и не должен смешиваться с игровыми и аналитическими runtime-контрактами.

## 8.4. Контракт email-отправки

В первой версии email используется как минимум для восстановления пароля.

Контракт почтовой интеграции должен обеспечивать:
- формирование письма на русском языке;
- вставку корректной одноразовой ссылки;
- логирование неудач доставки на уровне приложения.

## 8.5. Контракт логирования

Любая критически важная интеграционная ошибка должна приводить к записи в `AuditLogEntry` или эквивалентную таблицу логов.

Минимальные случаи логирования:
- ошибка игрового вызова LLM;
- ошибка аналитического вызова LLM;
- ошибка разбора аналитического JSON;
- ошибка PDF-генерации;
- ошибка email-отправки;
- попытка запрещённого доступа;
- конфликт активного диалога.

## 9. Коды ошибок и управляемые финальные состояния

## 9.1. Минимальный набор runtime error codes

Для JSON-endpoint-ов первой версии должен существовать как минимум следующий набор кодов:
- `ok`;
- `empty_message`;
- `message_too_long`;
- `dialog_not_found`;
- `dialog_not_active`;
- `forbidden`;
- `duplicate_submission_blocked`;
- `active_dialog_exists`;
- `llm_error`;
- `analysis_failed`;
- `analysis_skipped_no_user_messages`.

Разрешается расширять набор кодов, если это не ломает обратную совместимость runtime-обработчиков.

## 9.2. Управляемые финальные состояния пользователя

В пользовательском интерфейсе первая версия должна уметь приводить пользователя к контролируемому финальному состоянию в следующих случаях:
- анализ выполнен и результат доступен;
- анализ не выполнялся из-за отсутствия пользовательских реплик;
- LLM-ошибка в игровом запросе;
- LLM-ошибка в аналитике;
- конфликт второго активного диалога;
- попытка открыть архивный или недоступный сценарий.

Ни один из этих сценариев не должен завершаться необработанным traceback в браузере.

## 10. Нормативные замечания по будущему `/api/v1`

Этот раздел не вводит обязательный внешний API первой версии, но фиксирует требования к совместимости.

## 10.1. Будущие контуры

Позже должно быть возможно добавить:
- `/api/v1/auth/...`;
- `/api/v1/games/...`;
- `/api/v1/dialogs/...`;
- `/api/v1/results/...`;
- `/api/v1/profile/...`.

## 10.2. Правило совместимости

Будущий внешний API должен использовать тот же сервисный слой, что и веб-интерфейс первой версии.

Недопустимо создавать отдельную параллельную реализацию бизнес-логики только для APK.

## 10.3. Что уже сейчас должно быть стабильным

Чтобы позже добавить APK без перелома, уже в первой версии должны быть стабильны:
- `public_id` диалога;
- `slug` игры и сценария;
- DTO сообщений;
- DTO состояния диалога;
- формат аналитического JSON;
- сервисы запуска сценария, отправки сообщения, завершения, анализа и сборки результата.
