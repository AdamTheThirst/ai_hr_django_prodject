# NON_FUNCTIONAL_REQUIREMENTS.md

## 1. Назначение документа

Этот документ фиксирует нефункциональные требования к веб-тренажёру обратной связи на Django.

Под нефункциональными требованиями в рамках проекта понимаются:
- архитектурные ограничения;
- требования к надёжности;
- требования к безопасности;
- требования к производительности и устойчивости интерфейса;
- требования к качеству данных;
- требования к сопровождаемости и расширяемости;
- требования к развёртыванию и эксплуатации.

Документ должен использоваться как обязательная инженерная рамка при реализации проекта.

---

## 2. Общие принципы

### NFR-GENERAL-001
Проект должен реализовываться как один Django-монолит.

### NFR-GENERAL-002
Первая версия должна быть ориентирована на веб-приложение и одновременно проектироваться так, чтобы в будущем тот же бэкенд можно было использовать для подключаемого Android/APK-клиента без переписывания ядра бизнес-логики.

### NFR-GENERAL-003
В первой версии не требуется отдельный публичный API для внешних клиентов, но внутренняя архитектура не должна этому препятствовать.

### NFR-GENERAL-004
Интерфейс должен строиться по принципу mobile first.

### NFR-GENERAL-005
Интерфейс должен оставаться аккуратным и пригодным для использования как на мобильных устройствах, так и на десктопе.

### NFR-GENERAL-006
Фронтенд первой версии должен быть реализован на Django templates с Bootstrap 5 и локальным CSS.

### NFR-GENERAL-007
JavaScript в проекте должен использоваться только там, где он действительно необходим для пользовательского сценария.

### NFR-GENERAL-008
Для первой версии не должны использоваться тяжёлые SPA-подходы и отдельный фронтенд-фреймворк, если это не требуется явно зафиксированными сценариями.

### NFR-GENERAL-009
Визуальный стиль интерфейса должен быть минималистичным и опираться на принципы Material Design, но без обязательного подключения отдельной Material-библиотеки.

### NFR-GENERAL-010
Система должна использовать только русский язык интерфейса и контента в первой версии.

## 3. Архитектурные ограничения

### NFR-ARCH-001
Бизнес-логика не должна быть жёстко зашита во view-функции и шаблоны.

### NFR-ARCH-002
Ключевая логика диалога, завершения игры, запуска аналитики, работы с LLM, PDF-экспорта и прав доступа должна быть вынесена в отдельные сервисные слои или эквивалентные изолированные компоненты.

### NFR-ARCH-003
Архитектура должна позволять в будущем добавить API-слой вида `/api/v1/...` без переписывания моделей и ядра бизнес-логики.

### NFR-ARCH-004
Архитектура первой версии должна предусматривать будущую реализацию:
- личного кабинета;
- энциклопедии;
- пользовательской настройки таймера;
- голосовых сообщений;
- внешнего клиента на Android/APK.

### NFR-ARCH-005
Первая версия должна использовать SQLite как основную базу данных.

### NFR-ARCH-006
Несмотря на использование SQLite в первой версии, модель данных и ORM-слой должны проектироваться без привязки к SQLite-специфичным хакам, чтобы сохранить возможность последующего переноса на PostgreSQL.

### NFR-ARCH-007
Изменение структуры базы данных должно выполняться только через Django migrations.

### NFR-ARCH-008
Ключевые связи между сущностями должны реализовываться через корректные ForeignKey и иные штатные механизмы ORM, а не через логические связи без ссылочной целостности.

### NFR-ARCH-009
Мастер-данные проекта должны поддерживать архивацию вместо обязательного жёсткого удаления, если это требуется для сохранения целостности истории.

## 4. Требования к пользовательскому опыту

### NFR-UX-001
Игровой диалог должен происходить внутри одной страницы без полной перезагрузки интерфейса после каждого сообщения.

### NFR-UX-002
Пользователь не должен видеть технические перезагрузки страницы как основной способ обмена сообщениями в чате.

### NFR-UX-003
Во время ожидания ответа LLM интерфейс должен показывать понятный индикатор состояния «думаю...» в окне чата.

### NFR-UX-004
В первой версии не требуется стриминг ответа по токенам.

### NFR-UX-005
Блок `conditions` должен быть визуально закреплён в верхней части экрана и не должен теряться в длинной переписке.

### NFR-UX-006
Пользовательский интерфейс должен оставаться читаемым при длинных диалогах и длинных аналитических карточках.

### NFR-UX-007
Все интерактивные элементы должны иметь предсказуемое состояние при ожидании ответа сервера: активно, неактивно, заблокировано, обработано.

### NFR-UX-008
Система должна исключать визуально незаметную повторную отправку сообщения или повторное завершение диалога двойным кликом.

### NFR-UX-009
Кнопки интерфейса должны предусматривать место под иконку размером 24×24 px.

### NFR-UX-010
Аватар пользователя на основе первой буквы никнейма должен строиться на детерминированном светлом фоне, обеспечивающем читаемость чёрной буквы.

### NFR-UX-011
В интерфейсе не должны использоваться ядовитые, чрезмерно контрастные или плохо читаемые цветовые решения.

### NFR-UX-012
Пустые или ещё не реализованные разделы первой версии должны отображаться как честные заглушки, а не как сломанные или недоступные страницы.

## 5. Требования к производительности и отклику

### NFR-PERF-001
Система должна обеспечивать пользовательски приемлемую отзывчивость интерфейса для типовых сценариев первой версии на одном сервере с локальным хранением данных.

### NFR-PERF-002
Операции, не требующие обращения к LLM, не должны искусственно блокироваться длительными серверными процессами.

### NFR-PERF-003
Отправка сообщения в чат должна обрабатываться без полной перезагрузки страницы.

### NFR-PERF-004
Пока выполняется запрос к LLM, система должна предотвращать конкурентную отправку новых сообщений в тот же активный диалог.

### NFR-PERF-005
Пока выполняется завершение диалога и запуск анализа, система должна предотвращать повторный запуск того же завершения пользователем.

### NFR-PERF-006
Синхронный запуск аналитики в первой версии допустим, но интерфейс должен явно показывать пользователю состояние ожидания результата.

### NFR-PERF-007
Даже при синхронном анализе система не должна терять уже сохранённый транскрипт диалога.

### NFR-PERF-008
PDF-экспорт не должен изменять или повреждать исходные данные диалога и анализа.

### NFR-PERF-009
Система должна ограничивать длину пользовательского сообщения серверной валидацией согласно текущей настройке, по умолчанию 2500 символов.

### NFR-PERF-010
Система должна поддерживать раздельные настройки ограничений для игровых ответов и аналитических ответов LLM, управляемые супер-администратором.

## 6. Надёжность и отказоустойчивость

### NFR-REL-001
Система должна обеспечивать правило «один пользователь — один активный диалог одновременно» без состояния гонки между параллельными запросами.

### NFR-REL-002
Если пользователь завершил диалог вручную, система должна переводить диалог в финальное состояние один раз и без дублирования результатов.

### NFR-REL-003
Если диалог завершён по таймеру, система должна корректно блокировать дальнейший ввод, показать уведомление об истечении времени и перевести пользователя к анализу.

### NFR-REL-004
Если пользователь покинул страницу, система должна пытаться завершить диалог через механизм сигнализации со стороны браузера и иметь серверный запасной механизм добивания незавершённой сессии по таймауту неактивности.

### NFR-REL-005
При аварийном или неполном завершении диалог должен получать статус «не завершён», если именно такой результат соответствует согласованной логике.

### NFR-REL-006
Аналитика не должна запускаться для диалогов, в которых отсутствует хотя бы одна пользовательская реплика.

### NFR-REL-007
Если хотя бы одна пользовательская реплика существует, аналитика должна запускаться даже для незавершённого диалога.

### NFR-REL-008
Сохранение диалога, сообщений и аналитических результатов должно быть организовано так, чтобы частичный сбой не приводил к логически невозможным состояниям.

### NFR-REL-009
Система должна быть устойчива к повторной доставке одних и тех же запросов завершения диалога.

### NFR-REL-010
При ошибке внешнего LLM-вызова система должна:
- показать пользователю дружелюбное сообщение;
- выполнить предусмотренную стратегию повтора, если она разрешена текущей логикой;
- записать ошибку в технический лог.

### NFR-REL-011
При получении невалидного JSON от аналитического промта система должна корректно обрабатывать ошибку, а не падать необработанным исключением.

### NFR-REL-012
Обработка невалидного JSON аналитики должна включать логирование инцидента и предсказуемую ветку восстановления или отказа.

### NFR-REL-013
Временные служебные состояния диалога и анализа должны быть явно различимы на уровне данных и кода.

## 7. Целостность и качество данных

### NFR-DATA-001
Система должна хранить данные в нормализованной структуре, достаточной для поддержки игр, сценариев, промтов, сообщений, результатов анализа, настроек, UI-текстов и логов.

### NFR-DATA-002
Данные диалогов и аналитики не должны зависеть от удаления мастер-данных таким образом, чтобы история становилась нечитаемой или логически повреждённой.

### NFR-DATA-003
Итоговая сумма баллов должна вычисляться на лету из сохранённых результатов по критериям и не должна храниться отдельным дублирующим полем как обязательный источник истины.

### NFR-DATA-004
Система должна сохранять каждый аналитический результат отдельно вместе с текстом и целочисленным баллом.

### NFR-DATA-005
Система должна сохранять статус завершения диалога отдельно от содержания диалога.

### NFR-DATA-006
Система должна хранить редактируемые UI-тексты и системные настройки в виде данных, а не только как жёстко прошитые строки в коде.

### NFR-DATA-007
Система должна хранить LLM-параметры как управляемые настройки, доступные супер-администратору.

### NFR-DATA-008
Система должна хранить изображения и иные медиа локально в `MEDIA_ROOT` в первой версии.

### NFR-DATA-009
Система должна поддерживать переиспользование одного медиа-ресурса в разных сценариях, если это требуется логикой контента.

### NFR-DATA-010
При замене медиа-ресурсов система должна сохранять историю версий в пределах согласованной модели данных.

### NFR-DATA-011
Система должна разделять прикладные сущности и технические журналы так, чтобы логи не подменяли бизнес-данные.

---

## 8. Безопасность

### NFR-SEC-001
Система должна использовать стандартную безопасную аутентификацию Django, включая штатное хеширование паролей.

### NFR-SEC-002
Система должна использовать стандартные безопасные механизмы Django для работы с сессиями пользователя.

### NFR-SEC-003
Система должна использовать CSRF-защиту для пользовательских форм и чувствительных запросов браузерного интерфейса.

### NFR-SEC-004
Система должна использовать безопасный стандартный поток восстановления пароля с токенами ограниченного действия.

### NFR-SEC-005
Система должна обеспечивать ролевую модель доступа и проверку прав как минимум на уровне представлений, форм и прикладной логики.

### NFR-SEC-006
Система не должна позволять обычному пользователю выполнять административные действия путём прямого обращения к URL или подмены параметров запроса.

### NFR-SEC-007
Система не должна позволять администратору редактировать объекты, которые ему не принадлежат, если такое действие не разрешено его ролью.

### NFR-SEC-008
Система должна валидировать пользовательский ввод на стороне сервера независимо от клиентских ограничений.

### NFR-SEC-009
Система должна ограничивать длину пользовательских сообщений и иных критичных полей на стороне сервера.

### NFR-SEC-010
Система должна использовать базовую защиту от слишком частых попыток входа.

### NFR-SEC-011
Система должна использовать базовую защиту от слишком частой регистрации, если такая операция доступна публично.

### NFR-SEC-012
Система должна использовать базовую защиту от слишком частой отправки сообщений в чат.

### NFR-SEC-013
Механизмы ограничения частоты действий не должны ломать типовой пользовательский сценарий, но должны предотвращать очевидный спам и повторные запросы.

### NFR-SEC-014
Система не должна раскрывать пользователю внутренние стек-трейсы, системные промты, секреты конфигурации или иные внутренние технические детали через интерфейс ошибок.

### NFR-SEC-015
Технические логи не должны сохранять пароли, токены восстановления, секретные ключи и иные чувствительные секреты в открытом виде.

### NFR-SEC-016
При работе с LLM система должна относиться к содержимому пользовательского диалога как к недоверенному вводу и не должна позволять ему ломать управляющую логику приложения.

### NFR-SEC-017
Загрузка медиа-файлов через административный интерфейс должна сопровождаться серверной валидацией типа и размера файла в пределах согласованной реализации.

### NFR-SEC-018
При выходе пользователя из системы авторизованная сессия должна завершаться корректно.

## 9. Конфиденциальность и работа с персональными данными

### NFR-PRIV-001
Система должна хранить email пользователя как часть учётной записи, необходимой для входа, связи и восстановления пароля.

### NFR-PRIV-002
Система должна хранить никнейм как отображаемое имя пользователя.

### NFR-PRIV-003
В первой версии должна быть предусмотрена архитектурная возможность последующего добавления пользовательского согласия на обработку данных, но обязательный отдельный пользовательский сценарий согласия в текущем объёме не требуется.

### NFR-PRIV-004
При логировании технических событий система должна по возможности ограничивать объём персональных данных в логах до необходимого минимума.

### NFR-PRIV-005
Экспорт результатов в PDF должен включать только тот объём пользовательских данных, который соответствует согласованной структуре отчёта.

## 10. Сопровождаемость и расширяемость

### NFR-MAINT-001
Структура проекта должна быть понятной и предсказуемой для сопровождения и последующей генерации кода на основе спецификаций.

### NFR-MAINT-002
Система не должна опираться на скрытые магические значения там, где согласовано управление через админку или настройки.

### NFR-MAINT-003
Редактируемые тексты интерфейса, параметры LLM и контент игр должны храниться в управляемых сущностях и не требовать изменения исходного кода для обычной операционной работы.

### NFR-MAINT-004
Система должна позволять добавлять новые игры, сценарии и аналитические промты без изменения общей архитектуры приложения.

### NFR-MAINT-005
Система должна позволять увеличивать количество аналитических промтов для игры без изменений формата вывода результатов.

### NFR-MAINT-006
Система должна позволять заменять или уточнять системные промты без перепроектирования хранения диалогов и аналитики.

### NFR-MAINT-007
Код проекта должен быть пригоден для дальнейшего подключения API-слоя и мобильного клиента.

### NFR-MAINT-008
Развёртывание первой версии должно быть возможно без Docker, через стандартный `venv`.

### NFR-MAINT-009
Отсутствие Docker в первой версии не должно делать невозможным его последующее добавление.

---

## 11. Логирование и наблюдаемость

### NFR-OBS-001
Система должна логировать критичные технические события в отдельную таблицу базы данных.

### NFR-OBS-002
Система должна логировать как минимум следующие классы событий:
- ошибки LLM-запросов;
- ошибки разбора аналитического JSON;
- ошибки отправки писем восстановления пароля;
- ошибки прав доступа;
- ошибки загрузки медиа;
- попытки превышения допустимой частоты действий;
- системные исключения прикладного уровня.

### NFR-OBS-003
Логи должны содержать достаточно информации для диагностики причины сбоя без необходимости угадывать контекст вручную.

### NFR-OBS-004
Логи не должны подменять собой бизнес-историю диалога или аналитики.

### NFR-OBS-005
Административные действия, влияющие на контент и системные настройки, должны быть трассируемыми в пределах согласованной модели логирования.

## 12. Тестируемость и качество поставки

### NFR-TEST-001
Проект должен быть пригоден для автоматизированного тестирования критических сценариев.

### NFR-TEST-002
В проекте должны быть предусмотрены автоматизированные проверки как минимум для следующих направлений:
- регистрация и вход;
- восстановление пароля;
- запуск сценария;
- обмен сообщениями в диалоге;
- завершение диалога вручную;
- завершение диалога по таймеру;
- обработка незавершённого диалога;
- запуск аналитики;
- обработка невалидного JSON аналитики;
- ролевые ограничения доступа;
- PDF-экспорт.

### NFR-TEST-003
Тесты не должны зависеть от нестабильного ручного состояния базы данных.

### NFR-TEST-004
Для тестов должна быть предусмотрена возможность использовать контролируемые заглушки внешнего LLM-вызова.

### NFR-TEST-005
Seed-данные для локальной разработки не должны подменять тестовые фикстуры автоматически там, где это мешает воспроизводимости тестов.

## 13. Развёртывание и эксплуатация

### NFR-DEPLOY-001
Первая версия должна быть пригодна для локального запуска на компьютере разработчика через `venv`.

### NFR-DEPLOY-002
Первая версия должна быть пригодна для развёртывания на VPS без обязательного наличия Docker.

### NFR-DEPLOY-003
Для технической проверки система должна быть работоспособна без доменного имени; использование IP-адреса сервера допустимо.

### NFR-DEPLOY-004
Конфигурация окружения должна поддерживать разделение как минимум на локальное и серверное окружение.

### NFR-DEPLOY-005
Секреты окружения и параметры подключения к LLM не должны жёстко храниться в исходном коде.

### NFR-DEPLOY-006
Проект должен поддерживать локальное хранение медиа в `MEDIA_ROOT` рядом с проектом в первой версии.

### NFR-DEPLOY-007
Переход к другому типу хранения медиа в будущем не должен требовать переписывания доменной логики проекта.

## 14. Seed-данные первой версии

### NFR-SEED-001
В локальной и тестовой поставке проекта должны быть предусмотрены seed-данные для первичного запуска.

### NFR-SEED-002
Seed-данные первой версии должны включать как минимум:
- главного супер-администратора `superadmin`;
- тестового пользователя `testuser`;
- одну seed-игру;
- три seed-сценария;
- два аналитических промта из согласованного примера;
- системный промт для генерации служебных полей аналитики;
- набор UI-текстов по умолчанию.

### NFR-SEED-003
Указанные seed-учётные записи должны рассматриваться как данные для локального запуска и тестирования, а не как безопасные боевые учётные данные по умолчанию.

### NFR-SEED-004
Супер-администратор должен иметь возможность изменять данные этих seed-пользователей через административную логику, если это допускается моделью управления пользователями.
