# ARCHITECTURE.md

## 1. Назначение документа

Документ фиксирует целевую архитектуру первой веб-версии тренажёра обратной связи на Django.

Цель документа:
- описать структуру проекта на уровне модулей и приложений;
- зафиксировать границы ответственности компонентов;
- описать основные потоки данных и жизненный цикл диалога;
- определить точки расширения под будущий Android/APK-клиент, личный кабинет, энциклопедию и голосовые сообщения;
- задать такую архитектуру, по которой можно реализовать приложение без домысливания базовой структуры.

Документ не заменяет:
- `DATA_MODELS.md` — детальную схему сущностей и полей;
- `API_SPEC.md` — будущие контракты HTTP/API;
- `PROMPTS_SPEC.md` — устройство промтов и правила их исполнения.

---

## 2. Архитектурные принципы

### 2.1. Основной подход

Система реализуется как Django-монолит.

Это означает:
- один серверный проект;
- один набор Django apps внутри проекта;
- один веб-интерфейс на Django templates;
- одна база данных SQLite в первой версии;
- единая бизнес-логика, не дублируемая между вебом и будущими клиентами.

### 2.2. Ключевые принципы

1. Бизнес-логика не должна быть зашита во view-функции и шаблоны.
2. Ключевые сценарии должны проходить через сервисный слой.
3. Шаблоны отвечают за отображение, а не за принятие продуктовых решений.
4. Django Admin и пользовательская админка — это два разных интерфейса над одной моделью данных.
5. Архитектура первой версии должна позволять без перелома добавить `/api/v1/...` для будущего APK.
6. Все критичные связи реализуются через ForeignKey и другие штатные механизмы ORM.
7. Все изменения схемы выполняются через Django migrations.
8. Состояния диалогов и анализа должны быть явными и храниться в БД.
9. Любые удаляемые мастер-данные должны поддерживать архивацию, чтобы не ломать историю.

### 2.3. Что не используется в первой версии

В первой версии не используются:
- отдельный SPA-фронтенд;
- React/Vue и аналогичные тяжёлые клиентские фреймворки;
- обязательный публичный JSON API;
- фоновая очередь задач;
- стриминг токенов ответа LLM;
- Docker как обязательный способ запуска.

## 3. Общий архитектурный вид

Система состоит из пяти крупных слоёв:

1. Слой представления.
2. Слой прикладных сценариев.
3. Слой доменных моделей.
4. Слой инфраструктурных интеграций.
5. Слой административного управления.

### 3.1. Слой представления

Сюда входят:
- публичные веб-страницы на Django templates;
- формы регистрации, входа и восстановления пароля;
- главная страница с играми и кнопками сценариев;
- страница игрового чата;
- страница результатов анализа;
- экранные заглушки для личного кабинета и энциклопедии;
- отдельный внутренний административный интерфейс;
- стандартный Django Admin.

Этот слой:
- принимает HTTP-запросы;
- валидирует входные данные через формы/сериализаторы/служебные DTO;
- вызывает сервисный слой;
- отрисовывает HTML или возвращает JSON-ответ для частичного обновления страницы.

### 3.2. Слой прикладных сценариев

Это главный слой бизнес-логики.

Сюда входят сервисы, которые управляют:
- регистрацией и профилем пользователя;
- ролями и правами;
- запуском сценария;
- созданием и ведением диалога;
- отправкой сообщений в LLM;
- завершением диалога;
- синхронным запуском анализа;
- разбором и валидацией JSON анализа;
- экспортом PDF;
- управлением UI-текстами, системными настройками и параметрами LLM;
- загрузкой и переиспользованием медиа.

Этот слой не должен зависеть от конкретной HTML-разметки.

### 3.3. Слой доменных моделей

Слой моделей хранит нормализованные сущности:
- пользователи;
- профили и роли;
- игры;
- сценарии;
- сценарные медиа;
- игровые промты;
- аналитические промты;
- системные промты;
- диалоги;
- сообщения;
- результаты анализа;
- UI-тексты;
- системные настройки;
- технические логи.

### 3.4. Слой инфраструктуры

Слой инфраструктуры отвечает за работу с внешними и техническими ресурсами:
- OpenAI-compatible клиент для vLLM;
- email backend для восстановления пароля;
- файловое хранилище `MEDIA_ROOT`;
- PDF-генератор;
- логирование в таблицу БД;
- системные утилиты для времени, сериализации и валидации.

### 3.5. Слой административного управления

Этот слой состоит из двух интерфейсов:

1. Стандартный Django Admin.
2. Кастомный внутренний админ-интерфейс.

Django Admin нужен как:
- технический и резервный интерфейс;
- инструмент обзора моделей;
- быстрый способ обслуживания и ручной диагностики.

Кастомный админ-интерфейс нужен как:
- основной рабочий интерфейс супер-администраторов и администраторов;
- место управления играми, сценариями, промтами, UI-текстами и настройками;
- интерфейс, понятный не только разработчику.

## 4. Целевая структура Django-проекта

Ниже приводится рекомендуемая структура модулей. Названия можно менять только консистентно по всему проекту.

```text
project_root/
├── manage.py
├── config/
│   ├── settings/
│   │   ├── base.py
│   │   ├── local.py
│   │   └── production.py
│   ├── urls.py
│   ├── wsgi.py
│   └── asgi.py
├── apps/
│   ├── core/
│   ├── accounts/
│   ├── content/
│   ├── dialogs/
│   ├── analysis/
│   ├── platform_config/
│   ├── adminpanel/
│   ├── exports/
│   ├── auditlog/
│   └── integrations/
├── templates/
├── static/
├── media/
├── fixtures/
└── requirements/
```

### 4.1. `config/`

Содержит конфигурацию Django-проекта:
- settings по окружениям;
- корневой `urls.py`;
- WSGI/ASGI entrypoints.

### 4.2. `apps/core/`

Общие технические элементы проекта:
- базовые абстрактные модели;
- mixins;
- общие исключения;
- общие перечисления;
- общие утилиты;
- базовые сервисные контракты.

В `core` нельзя складывать предметную бизнес-логику конкретных модулей.

### 4.3. `apps/accounts/`

Отвечает за пользователей и доступ.

Внутри модуля:
- пользовательская модель или расширение стандартной модели Django;
- профиль пользователя;
- никнейм;
- email;
- аватарная буква и детерминированный цветовой ключ;
- роли и флаги доступа;
- регистрация;
- вход;
- выход;
- восстановление пароля;
- редактирование никнейма;
- seed-пользователи для dev-режима.

Отдельно здесь же должна жить логика ролей:
- обычный пользователь;
- администратор;
- назначенный супер-администратор;
- главный супер-администратор.

### 4.4. `apps/content/`

Отвечает за контент игры.

Внутри модуля:
- игры;
- сценарии;
- порядок сценариев на экране;
- текст `conditions`;
- стартовая реплика персонажа;
- игровые промты;
- аналитические промты;
- системные промты;
- медиа сценариев;
- архивирование и переиспользование медиа;
- принадлежность сущностей к администратору-создателю.

Логика правила:
- у игры много сценариев;
- у игры один набор аналитических промтов;
- сценарии одной игры используют этот общий набор.

### 4.5. `apps/dialogs/`

Отвечает за игровой диалог.

Внутри модуля:
- создание новой игровой сессии;
- проверка правила "один активный диалог на пользователя";
- хранение статусов диалога;
- хранение сообщений;
- отправка пользовательского сообщения;
- сохранение ответа ИИ;
- таймер;
- завершение диалога вручную;
- завершение по таймеру;
- завершение при покидании страницы;
- блокировки от повторной отправки.

Это центральный модуль пользовательского игрового цикла.

### 4.6. `apps/analysis/`

Отвечает за анализ завершённого диалога.

Внутри модуля:
- получение полного транскрипта диалога;
- выбор активного набора аналитических промтов для игры;
- синхронный последовательный запуск аналитики по каждому критерию;
- проверка JSON-ответов;
- retry/обработка ошибок по утверждённой логике;
- сохранение отдельных результатов анализа;
- вычисление суммарного балла на лету;
- подготовка карточек для экрана результата.

### 4.7. `apps/platform_config/`

Отвечает за глобальные настройки проекта.

Внутри модуля:
- UI-тексты;
- системные параметры;
- LLM-параметры;
- глобальный таймер по умолчанию;
- лимит длины пользовательского сообщения;
- лимиты длины игровых и аналитических ответов;
- пояснения к параметрам для супер-администратора;
- будущие фичи-флаги для личного кабинета, энциклопедии и голоса.

Этот модуль не хранит контент конкретной игры. Он хранит глобальные настройки платформы.

### 4.8. `apps/adminpanel/`

Отвечает за кастомный внутренний административный интерфейс.

Внутри модуля:
- административные страницы;
- формы управления контентом;
- формы управления ролями;
- формы управления UI-текстами и системными настройками;
- проверки прав доступа на уровне интерфейса;
- таблицы просмотра диалогов, результатов и логов.

### 4.9. `apps/exports/`

Отвечает за экспорт документов.

В первой версии:
- генерация PDF с результатами анализа.

### 4.10. `apps/auditlog/`

Отвечает за техническое логирование.

Внутри модуля:
- запись ошибок LLM;
- запись ошибок JSON-парсинга;
- запись аварийных состояний;
- запись административно значимых действий;
- запись системных событий, полезных для диагностики.

### 4.11. `apps/integrations/`

Отвечает за внешние интеграции.

Внутри модуля:
- клиент vLLM через OpenAI-compatible API;
- почтовая интеграция;
- служебные адаптеры к внешним сервисам.

Вся работа с внешним LLM должна идти через этот модуль, а не напрямую из view или модели.

## 5. Логическая схема взаимодействия модулей

### 5.1. Разрешённое направление зависимостей

Разрешённая схема зависимостей:

```text
views/templates/admin UI
        ↓
application services
        ↓
domain models + repositories/ORM access
        ↓
integrations / infrastructure
```

Шаблоны не должны напрямую вызывать LLM.

Модели не должны содержать внутри себя тяжёлую прикладную оркестрацию, например:
- запуск анализа;
- генерацию PDF;
- управление правами;
- разбор ответа внешней модели.

### 5.2. Взаимодействие ключевых модулей

#### `accounts` ↔ `adminpanel`
`adminpanel` использует `accounts` для управления ролями, доступами и seed-пользователями.

#### `content` ↔ `dialogs`
`dialogs` использует `content` для получения:
- игры;
- сценария;
- текста `conditions`;
- стартовой реплики персонажа;
- актуального игрового промта.

#### `dialogs` ↔ `integrations`
`dialogs` вызывает LLM-клиент для получения игровых ответов персонажа.

#### `dialogs` ↔ `analysis`
После завершения диалога `dialogs` передаёт управление в `analysis`, если в диалоге есть хотя бы одна пользовательская реплика.

#### `analysis` ↔ `content`
`analysis` получает из `content` набор аналитических промтов для игры.

#### `analysis` ↔ `integrations`
`analysis` вызывает LLM-клиент для каждого аналитического промта.

#### `analysis` ↔ `platform_config`
`analysis` получает из `platform_config` рабочие настройки LLM и лимиты.

#### `exports` ↔ `dialogs` / `analysis`
`exports` читает сохранённый диалог и результаты анализа и формирует PDF.

#### `auditlog` ↔ все сервисные модули
Все критичные ошибки и служебные события должны уметь логироваться в `auditlog`.

## 6. Веб-слой: структура интерфейсов

## 6.1. Публичный веб-интерфейс

Публичный слой состоит из следующих групп страниц:
- регистрация;
- вход;
- восстановление пароля;
- главная страница с играми и сценариями;
- страница чата;
- страница результатов;
- заглушка личного кабинета;
- заглушка энциклопедии.

### 6.2. Паттерн обновления страницы

Используется гибридный подход:
- обычная серверная отрисовка HTML для полноценных страниц;
- минимальный JavaScript для асинхронных действий внутри страницы чата и служебных переходов.

В первой версии JavaScript нужен для:
- отправки сообщения без полной перезагрузки;
- вывода индикатора "думаю...";
- автопрокрутки чата;
- обратного отсчёта таймера;
- обработки истечения времени;
- сигнала о покидании страницы;
- защиты от двойной отправки.

### 6.3. Частичные ответы сервера

Для чата допустимы два формата внутренних ответов:
- HTML-фрагменты для встраивания в DOM;
- компактный JSON для клиентского JS.

В первой версии следует выбрать один консистентный механизм и использовать его во всех чатовых действиях. Предпочтительный вариант — компактный JSON и клиентский рендер типовых блоков сообщений.

## 7. Логика жизненного цикла диалога

## 7.1. Состояния диалога

Для архитектуры первой версии фиксируются следующие состояния диалога:
- `active` — диалог открыт, пользователь может писать;
- `completed_manually` — диалог завершён кнопкой «Дай обратную связь»;
- `completed_by_timeout` — диалог завершён по таймеру;
- `aborted` — пользователь покинул страницу или сессия оборвалась;
- `analysis_skipped_no_user_messages` — завершение произошло без единой пользовательской реплики;
- `analysis_failed` — анализ попытались выполнить, но он завершился ошибкой;
- `results_ready` — анализ завершён, результаты доступны.

Точное хранение статусов и переходов должно быть закреплено в `DATA_MODELS.md`, но код не должен использовать неявные текстовые значения, разбросанные по проекту.

## 7.2. Создание диалога

При запуске сценария система должна:
1. проверить, что у пользователя нет другого активного диалога;
2. создать новую запись диалога;
3. связать её с пользователем, игрой и сценарием;
4. зафиксировать текущие эффективные настройки таймера;
5. сохранить стартовую реплику персонажа как первое сообщение assistant;
6. открыть страницу чата.

## 7.3. Общение в диалоге

При отправке сообщения система должна:
1. проверить, что диалог активен;
2. проверить длину сообщения;
3. защититься от повторной отправки;
4. сохранить пользовательское сообщение;
5. собрать полный транскрипт диалога;
6. получить игровой промт сценария/игры;
7. вызвать LLM;
8. сохранить ответ assistant;
9. вернуть обновление чата на текущую страницу.

## 7.4. Завершение вручную

При нажатии кнопки «Дай обратную связь» система должна:
1. заблокировать повторное завершение;
2. перевести диалог в финальный статус завершения;
3. проверить, есть ли хотя бы одна пользовательская реплика;
4. если пользовательских реплик нет — не запускать анализ и перевести диалог в `analysis_skipped_no_user_messages`;
5. если пользовательские реплики есть — синхронно запустить анализ;
6. после завершения анализа открыть страницу результатов.

## 7.5. Завершение по таймеру

При истечении времени система должна:
1. показать UI-сообщение об истечении времени;
2. выдержать 3 секунды;
3. заблокировать дальнейший ввод;
4. зафиксировать статус `completed_by_timeout`;
5. проверить наличие пользовательских реплик;
6. при наличии реплик запустить анализ;
7. перевести пользователя на экран результата.

## 7.6. Завершение при уходе со страницы

При покидании страницы система должна использовать двухступенчатую схему:

1. Клиент пытается отправить финальный сигнал через `sendBeacon`/эквивалент.
2. Если браузерный сигнал не дошёл, сервер завершает зависший активный диалог запасным механизмом по таймауту неактивности.

Если в диалоге нет ни одной пользовательской реплики:
- анализ не запускается.

Если есть хотя бы одна пользовательская реплика:
- диалог получает статус `aborted`;
- анализ запускается;
- результаты сохраняются;
- пользователю при следующем просмотре должна быть доступна информация, что диалог был не завершён.

## 8. Логика жизненного цикла анализа

## 8.1. Общий принцип

Анализ работает синхронно и запускается только после завершения диалога.

Один аналитический промт = один критерий.

Для одной игры может существовать произвольное количество аналитических промтов.

### 8.2. Шаги анализа

Сервис анализа должен:
1. получить диалог;
2. убедиться, что есть минимум одна пользовательская реплика;
3. получить игру диалога;
4. получить активный набор аналитических промтов этой игры в заданном порядке;
5. для каждого промта:
   - собрать запрос к LLM;
   - отправить запрос;
   - получить ответ;
   - проверить валидность JSON;
   - проверить наличие полей `rating` и `text`;
   - проверить допустимость типов и диапазонов;
   - сохранить отдельный результат;
6. после прохода по всем критериям пометить анализ завершённым;
7. подготовить данные для экрана результата.

### 8.3. Работа с ошибками анализа

Если LLM возвращает ошибку или невалидный JSON, система должна:
- сохранить технический лог;
- применить утверждённую стратегию повторной попытки, если она настроена;
- не допустить необработанного падения страницы;
- зафиксировать итоговое состояние анализа.

### 8.4. Расчёт суммы

Суммарный балл не хранится как отдельный обязательный источник истины.

Он вычисляется в сервисном слое на основе:
- количества сохранённых результатов;
- суммы `rating`;
- теоретического максимума по числу критериев.

## 9. Архитектура работы с LLM

## 9.1. Общий подход

В проекте используется один и тот же внешний LLM-бэкенд для:
- игрового диалога;
- аналитики;
- вспомогательных системных задач, например генерации служебных alias/header/comment.

Модель одна и та же, различаются только промты и режимы вызова.

### 9.2. LLM-клиент

Работа с моделью должна быть инкапсулирована в отдельном клиенте/адаптере внутри `apps/integrations/`.

Этот адаптер должен:
- принимать параметры подключения;
- принимать модель;
- принимать `temperature`, `top_p`, `max_tokens`;
- формировать OpenAI-compatible запрос;
- возвращать унифицированный ответ сервисному слою;
- пробрасывать нормализованные ошибки.

### 9.3. Источник параметров

Параметры LLM не должны быть жёстко прошиты только в коде.

Эффективные значения берутся из:
- глобальных настроек платформы;
- активных промтов;
- валидных ограничений, доступных супер-администратору.

### 9.4. Разделение контекстов

Архитектурно должны быть разделены:
- игровой контекст;
- аналитический контекст;
- служебный контекст для генерации внутренних полей.

Даже если физически используется один и тот же клиент, на уровне сервисов это разные сценарии вызова.

---

## 10. Архитектура прав доступа

## 10.1. Общий принцип

Права должны проверяться в двух местах:
- на уровне маршрутов/представлений;
- на уровне сервисов.

Одной только проверки в шаблоне недостаточно.

### 10.2. Модель владения контентом

Игры, сценарии, промты и часть связанных сущностей должны иметь признак автора/создателя.

Это нужно для правила:
- обычный администратор управляет только своими объектами;
- назначенный супер-администратор управляет всеми объектами;
- главный супер-администратор управляет всеми объектами и ролями верхнего уровня.

### 10.3. Главный супер-администратор

Архитектурно в системе должен существовать один главный супер-администратор.

Его полномочия включают:
- создание администраторов;
- назначение и отзыв супер-администраторов;
- полный административный контроль.

### 10.4. Назначенный супер-администратор

Может управлять:
- UI-текстами;
- системными настройками;
- LLM-параметрами;
- промтами;
- любыми играми и сценариями.

Не может:
- создавать новых администраторов.

## 11. Архитектура хранения медиа

### 11.1. Общий подход

Все медиа первой версии хранятся локально в `MEDIA_ROOT`.

### 11.2. Требования к организации

Архитектура должна поддерживать:
- загрузку файла через админский интерфейс;
- связь одного файла с несколькими сценариями;
- сохранение истории версий;
- мягкую архивацию старых файлов, а не обязательное физическое удаление.

### 11.3. Использование в сценарии

Сценарий должен ссылаться на медиа как на отдельную сущность, а не хранить только сырую строку пути к файлу без модели.

---

## 12. Архитектура UI-текстов и системных настроек

## 12.1. UI-тексты

Все важные тексты интерфейса, которые по условиям проекта должны меняться через админку, должны храниться как данные.

Примеры:
- «Время диалога истекло»;
- тексты заглушек;
- подписи кнопок;
- служебные уведомления.

### 12.2. Системные настройки

Глобальные настройки платформы хранятся отдельно от UI-текстов.

Сюда входят:
- таймер по умолчанию;
- лимит длины сообщения пользователя;
- лимиты для игровых и аналитических ответов;
- параметры LLM;
- служебные флаги.

### 12.3. Эффективные настройки диалога

На момент старта диалога часть глобальных значений должна сниматься в конкретный диалог как эффективные параметры сессии.

Это нужно, чтобы уже начатый диалог не менял своё поведение из-за последующего редактирования глобальных настроек.

---

## 13. Архитектура экспорта PDF

PDF не должен зависеть от текущего состояния интерфейса браузера.

Он должен строиться из сохранённых данных:
- диалог;
- сценарий;
- игра;
- результаты анализа.

Сервис PDF должен:
- принимать идентификатор результата/диалога;
- собирать данные;
- формировать документ с согласованной структурой;
- отдавать файл пользователю без изменения исходной БД.

---

## 14. Техническое логирование

Логирование должно быть выделено в отдельный модуль и отдельные сущности БД.

Нужно логировать как минимум:
- ошибки LLM-запросов;
- ошибки парсинга JSON анализа;
- ошибки PDF-экспорта;
- ошибки отправки email;
- подозрительные или конфликтные состояния диалога;
- критичные административные действия.

Технические логи не должны смешиваться с пользовательскими результатами.

---

## 15. Seed-данные и окружения

## 15.1. Seed-данные для разработки

В dev-режиме должны существовать seed-данные:
- главный супер-администратор `superadmin / 123456`;
- тестовый пользователь `testuser / 123`;
- одна игра;
- три сценария-кнопки внутри этой игры;
- один персонаж Алексей во всех трёх сценариях;
- два аналитических промта из согласованных примеров;
- системный промт для генерации alias/header/comment;
- базовые UI-тексты.

## 15.2. Важное ограничение

Эти seed-учётки допустимы только как стартовые данные для локального/dev-запуска и первичной настройки. В production они должны подлежать смене через административный интерфейс.

## 16. Подготовка к будущему APK и API

### 16.1. Что закладывается сразу

В первую версию закладывается следующее:
- сервисный слой без привязки к HTML;
- чёткие доменные сущности;
- единый LLM-адаптер;
- явные статусы диалога и анализа;
- возможность повторного использования тех же сервисов из будущего API.

### 16.2. Что не делается в первой версии

В первой версии не создаётся полноценный внешний API-контур для APK.

### 16.3. Что должно быть легко добавлено позже

Позже должно быть возможно добавить:
- `/api/v1/auth/...`;
- `/api/v1/games/...`;
- `/api/v1/dialogs/...`;
- `/api/v1/results/...`;
- `/api/v1/profile/...`.

При этом ядро модулей `accounts`, `content`, `dialogs`, `analysis`, `platform_config` и `integrations` не должно переписываться заново.

## 18. Архитектурные антипаттерны, которые запрещены

В реализации запрещено:
- хранить ключевые UI-тексты только в шаблонах, если они должны редактироваться через админку;
- вызывать LLM напрямую из шаблонов, моделей или случайных утилит;
- держать проверки ролей только на фронтенде;
- дублировать бизнес-логику одновременно во view и JS;
- хранить игры, сценарии и промты без нормальных связей ORM;
- рассчитывать итог анализа из невалидных или частично несохранённых данных без явного статуса;
- жёстко связывать архитектуру только с веб-интерфейсом так, чтобы потом нельзя было добавить APK-клиент.

## 19. Итоговая архитектурная схема

Система должна реализовываться как Django-монолит со следующим ядром:

- `accounts` — пользователи, роли, вход, восстановление пароля;
- `content` — игры, сценарии, промты, медиа;
- `dialogs` — игровой чат, сообщения, таймер, завершение;
- `analysis` — аналитика, JSON-валидация, результаты;
- `platform_config` — UI-тексты, глобальные настройки, параметры LLM;
- `adminpanel` — пользовательская админка;
- `exports` — PDF;
- `auditlog` — технические журналы;
- `integrations` — vLLM и email.

Именно эта структура должна считаться базовой целевой архитектурой первой версии.
